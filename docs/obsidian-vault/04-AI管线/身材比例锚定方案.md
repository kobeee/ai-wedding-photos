# 身材比例锚定方案

## 问题

AI 生双人婚纱照时，身高差和体型对比主要靠模型猜测。单人参考图可以帮助保脸、保气质，但对“双人相对比例”帮助有限。

当前状态：
- 上传阶段只做格式和大小校验，不采集双人比例相关参考
- 前端文案只提示“正面、侧面、全身”，没有专门引导用户上传双人全身合照
- 生图 prompt 里的 `preserve body shape` 约束过弱
- VLM 质检会看 `body proportions`，但属于事后发现，不是事前锚定

## 修订后的核心判断

这件事值得做，但第一版应定义为：

**弱比例锚定，不是强约束锚定。**

也就是说：
- 系统优先利用双人全身合照提升“双人相对身材比例”的稳定性
- 没有合照或合照不可用时，必须无损退回现有 bride + groom 单人参考模式
- 第一版不宣称“自动安全识别有效合照”，只宣称“前端强引导 + 参考图前置 + prompt 弱锚定”

## 为什么用合照，而不是填数字

AI 生图更接近“看图说话”，不是“看数字说话”。

直接写：
- 男 175cm
- 女 163cm

模型未必稳定理解。

但如果给它一张两人站在一起的全身日常照，模型更容易直接感知：
- 相对身高差
- 整体体型对比
- 肩宽、腿长、骨架等视觉关系

因此第一版采用：

**上传 1 张双人全身合照作为额外参考图，并尽量放在参考图列表第一位。**

## 为什么不选其他方案

| 方案 | 否决原因 |
|------|----------|
| 让用户填身高体重 | 违反“零决策压力”，且数字写进 prompt 不稳定 |
| 从单人照自动推绝对身高 | 没有共同参照物，意义有限 |
| 让用户二次确认或手动微调比例 | 增加步骤，增加决策压力 |
| 上来就做骨骼检测 / 身高像素计算 | 第一版过重，投入产出比低 |

## 产品策略

### Upload 页面

- 改为三段式布局：双人合照区在上，新郎 / 新娘个人照在下
- 双人合照区最多 1 张，文案明确写“建议上传双人全身日常合照”
- 新郎 / 新娘照片继续要求包含全身照和面部清晰照
- 合照是**强提示但不强制**
- 缺少合照时，下一步按钮**不 disabled**

原因：
- 合照能提升效果，但不应卡死流程
- 比例锚定是增强项，不是流程前置门槛

## 工程策略

### Phase 0：前置条件

在做比例锚定之前，先补齐上传链路。

必须先完成：
- Upload 页面真实调用 `/api/upload`
- 前端持久化 `user_id` 和 `session_token`
- 后续 `/api/makeup`、`/api/generate` 带 `Authorization: Bearer <session_token>`

没有这一步，比例锚定方案无法真正落地。

### Phase 1：弱比例锚定

在上传链路打通后，再做以下改动。

#### 1. 上传角色扩展

- `/api/upload` 的 role 白名单新增 `couple`
- sidecar metadata 保留 `role`
- 前端上传时：
  - 合照区 → `role="couple"`
  - 新郎区 → `role="groom"`
  - 新娘区 → `role="bride"`

#### 2. 参考图选择策略

`reference_selector.py` 改为：
1. 如果存在 `couple` 候选图且通过基础质量筛选，则放第一位
2. bride 最佳单人照
3. groom 最佳单人照
4. 剩余名额按质量分补齐

并同步：
- `MAX_TOTAL` 从 3 调到 4
- role 解析支持 `couple`
- 统计维度同步支持 `couple`
- 补对应测试

### 关于“有效合照”的真实表述

第一版**不做人数检测，不做骨骼检测，不做 VLM 合照确认**。

因此第一版里的 `couple` 图只是：

**由用户显式上传到合照区，并通过基础质量筛选的弱候选锚点。**

这意味着：
- 系统会优先尝试使用它
- 但不能宣称它一定是真正有效的“双人比例锚”
- 文档中不能写成“自动安全降级”或“自动识别有效合照”

更准确的说法是：

**第一版依赖前端引导 + 参考图前置 + prompt 条件句做软兜底。**

#### 3. Prompt 锚定指令

当存在 `couple` 候选锚点时，`prompt_assembler.py` 追加弱锚定指令：

> The first reference image is a full-body couple photo. When both people are visible, preserve the relative height difference and overall build shown there, while keeping the final composition natural and photographic.

这段文案的设计原则：
- 用 `relative height difference`，不用 `exact height ratio`
- 用 `When both people are visible` 作为条件句
- 保留模型的构图和透视弹性，避免把比例约束写死

## 明确不做的事

第一版不做：
- 不加 body_reference 字段到 `CreativeBrief`
- 不改 `GenerateRequest` schema
- 不改 `nano_banana.py` 的接口形态
- 不做人数识别
- 不做骨骼检测
- 不做 VLM 合照有效性确认

## 数据流

```text
前端 Upload
  ├── coupleFiles[0]  → POST /api/upload (role="couple")
  ├── maleFiles       → POST /api/upload (role="groom")
  └── femaleFiles     → POST /api/upload (role="bride")
        ↓
upload metadata sidecar
        ↓
reference_selector.py
  └── 优先尝试前置 couple 候选图 → bride 最佳 → groom 最佳 → 辅助图
        ↓
prompt_assembler.py
  └── 仅在存在 couple 候选锚点时追加弱比例锚定文案
        ↓
nano_banana → Gemini API
```

## 风险与真实边界

1. **模型未必稳定尊重第一张参考图的比例关系**
   这需要联调验证。第一版目标是“提高命中率”，不是“绝对锁定比例”。

2. **误标风险不能被说成自动安全降级**
   如果用户把单人照误传到合照区，第一版没有人数校验，只能依赖前端引导和 prompt 条件句做软兜底。

3. **弱锚定提升上限，但不保证下限**
   这是一种轻量增强，不是严格控制系统。

4. **后续如验证收益明显，再考虑补最小校验信号**
   例如 VLM 轻量确认“是否为双人全身照”，但这不属于第一版范围。

## 测试要求

至少补以下测试：
- `couple` role 能被 upload 和 metadata 正确识别
- `reference_selector` 能把 `couple` 候选图前置
- 没有 `couple` 图时，行为保持当前 bride + groom 逻辑
- prompt 只在存在 `couple` 候选锚点时注入弱锚定文案
