# 双模型协作

> [[Nano-Banana-Pro]] 与 [[GPT-Image-1.5]] 的分工与协作策略。

---

## 协作理念

**一个负责"画"，一个负责"修"**。

两个模型各有所长，通过精心设计的分工实现超越单模型的生成质量：

- [[Nano-Banana-Pro]]：画质担当 — 高分辨率渲染 + 面部 ID 保持
- [[GPT-Image-1.5]]：情感担当 — 面部表情控制 + 精细局部编辑

---

## 分工矩阵

| 任务 | 主模型 | 原因 |
|------|--------|------|
| 4K 底图生成 | [[Nano-Banana-Pro]] | 画质强，支持多图参考保持 ID |
| 物理错误修复（手指/穿模） | [[Nano-Banana-Pro]] | 物理准确性好 |
| 表情/情绪修复 | [[GPT-Image-1.5]] | 面部表情控制精准 |
| 面部精修 | [[GPT-Image-1.5]] | 局部编辑能力强 |
| AI 试妆预览 | [[Nano-Banana-Pro]] | 需要 ID 保持 |
| 魔法笔刷微调 | [[GPT-Image-1.5]] | 交互式编辑 |

---

## 协作流程

```
Phase 1: 底图生成
┌───────────────────────────┐
│  Nano Banana Pro          │
│  - 输入: 用户照片 + Prompt  │
│  - 输出: 4K 底图           │
│  - 模式: 图生图(多图参考)   │
└───────────┬───────────────┘
            │
Phase 2: VLM 质检
┌───────────┴───────────────┐
│  VLM 视觉反思              │
│  → 检出物理错误 / 情绪错误  │
│  参见 [[VLM质检管线]]       │
└───────────┬───────────────┘
            │
Phase 3: 分流修复
┌───────────┴───────────────┐
│                           │
│  物理错误?                 │
│  ┌─────────────────┐      │
│  │ Nano Banana Pro │      │
│  │ 局部重绘        │      │
│  └─────────────────┘      │
│                           │
│  情绪错误?                 │
│  ┌─────────────────┐      │
│  │ GPT-Image-1.5   │      │
│  │ 遮罩式 I2I 重绘 │      │
│  └─────────────────┘      │
└───────────┬───────────────┘
            │
Phase 4: 回到 VLM 复检
(最多循环 3 次)
```

---

## 关键决策：为什么不用单模型？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 只用 Nano Banana Pro | 画质好、ID 保持强 | 表情控制弱，面部僵硬 |
| 只用 GPT-Image-1.5 | 表情自然、编辑灵活 | 画质不及 Nano，ID 保持弱 |
| **双模型协作** | **兼得两者优点** | 调用链路长，成本更高 |

选择双模型方案的核心原因：**婚纱照对面部表情的要求极高，同时画质不能有妥协**。

## 成本控制

| 模型 | 单次调用成本（预估） | 场景 |
|------|---------------------|------|
| [[Nano-Banana-Pro]] | 约 ¥0.1-0.3/张 | 底图 + 物理修复 |
| [[GPT-Image-1.5]] | 约 ¥0.2-0.5/次 | 情绪修复 |

每组婚纱照预估成本：
- 正常流程（1次生成 + 1次修复）：¥0.5-1.0
- 最差情况（3轮修复循环）：¥1.5-3.0
- 20 张套餐总成本：¥10-60

利润空间充足，参见 [[商业模式]]。

---

## 容错策略

| 场景 | 处理方式 |
|------|----------|
| Nano API 不可用 | 切换到 GPT-Image-1.5 全流程 |
| GPT API 不可用 | Nano 尝试情绪修复（质量降级） |
| 双模型均不可用 | 返回 503，前端展示维护提示 |
| 3 轮修复仍不合格 | 强制交付 + 标注质量分数 |

## 后续优化方向

1. 引入更多专项模型（背景生成、服饰渲染）
2. VLM 质检 Prompt 持续优化（减少误判）
3. 建立修复效果评估数据集
4. 探索模型蒸馏，降低推理成本
