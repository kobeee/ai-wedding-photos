# 双模型协作

> [[Nano-Banana-Pro]] 与 [[GPT-Image-1.5]] 的分工与协作策略。

> 2026-03-25 更新：当前生产实现已切到 **Nano Banana 全修复主链路**。GPT-Image-1.5 保留服务封装与配置开关，但默认关闭，待后续补齐遮罩与效果评估后再决定是否恢复。

---

## 协作理念

**一个负责"画"，一个负责"修"**。

两个模型各有所长，通过精心设计的分工实现超越单模型的生成质量：

- [[Nano-Banana-Pro]]：画质担当 — 高分辨率渲染 + 面部 ID 保持
- [[GPT-Image-1.5]]：情感担当 — 面部表情控制 + 精细局部编辑

---

## 分工矩阵

| 任务 | 主模型 | 原因 |
|------|--------|------|
| 4K 底图生成 | [[Nano-Banana-Pro]] | 画质强，支持多图参考保持 ID |
| 物理错误修复（手指/穿模） | [[Nano-Banana-Pro]] | 物理准确性好 |
| 表情/情绪修复 | [[Nano-Banana-Pro]] | 当前实现统一走 Nano 修复，减少链路和成本 |
| 面部精修 | [[GPT-Image-1.5]] | 局部编辑能力强 |
| AI 试妆预览 | [[Nano-Banana-Pro]] | 需要 ID 保持 |
| 魔法笔刷微调 | [[GPT-Image-1.5]] | 交互式编辑 |

```

            │

│  VLM 视觉反思              │

│  参见 [[VLM质检管线]]       │
└───────────┬───────────────┘
            │
Phase 3: 修复
┌───────────┴───────────────┐
│                           │
│  物理错误 / 情绪错误?      │
│  ┌─────────────────┐      │
│  │ Nano Banana Pro │      │

            │
Phase 4: 回到 VLM 复检
```

## 当前运行态：为什么先用单模型修复？

当前代码里，GPT 修复链路虽然保留了 API 封装，但还没有真正做到“遮罩式局部重绘”：

- `services/gpt_image.py` 支持 `mask_path`
- 主链路调用没有传 mask
- VLM 质检目前也不产出可直接用于 edit 的区域掩码

在这种前提下，GPT 更像“全图盲修”，还不能证明比 Nano 的多图修复更值。

所以当前运行态采用：

- 底图生成：[[Nano-Banana-Pro]]
- physical repair：[[Nano-Banana-Pro]]
- emotional repair：[[Nano-Banana-Pro]]
- GPT-Image-1.5：保留接口，默认关闭，作为后续实验开关

---

## 关键决策：为什么仍保留 GPT 预留位？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 只用 Nano Banana Pro | 链路短、成本低、ID 保持强 | 表情专项能力仍待实测验证 |
| 只用 GPT-Image-1.5 | 表情自然、编辑灵活 | 画质不及 Nano，ID 保持弱 |
| **双模型协作** | **理论上兼得两者优点** | 需要遮罩与实测数据支撑，否则链路长、成本更高 |

保留 GPT 的原因不是“今天就要用”，而是：

1. 后续如果补齐 face mask / region detection，可以快速重开实验。
2. 当前服务封装、配置和健康检查都还在，恢复成本很低。
3. 是否恢复必须由实测修复成功率、延迟和单张成本共同决定，而不是凭感觉。

## 成本控制

| 模型 | 单次调用成本（预估） | 场景 |
|------|---------------------|------|
| [[Nano-Banana-Pro]] | 约 ¥0.1-0.3/张 | 底图 + 物理修复 |
| [[GPT-Image-1.5]] | 约 ¥0.2-0.5/次 | 预留专项实验 |

每组婚纱照预估成本：
- 正常流程（1次生成 + 1次修复）：¥0.5-1.0
- 最差情况（3轮修复循环）：¥1.5-3.0
- 20 张套餐总成本：¥10-60

利润空间充足，参见 [[商业模式]]。

## 容错策略

| 场景 | 处理方式 |
|------|----------|
| Nano API 不可用 | 当前主链路不可用，返回 503 |
| GPT API 不可用 | 当前默认不受影响；仅影响后续实验开关 |
| 双模型均不可用 | 返回 503，前端展示维护提示 |
| 3 轮修复仍不合格 | 强制交付 + 标注质量分数 |

## 后续优化方向

1. 引入更多专项模型（背景生成、服饰渲染）
2. VLM 质检 Prompt 持续优化（减少误判）
3. 建立修复效果评估数据集，比较 Nano-only 与 GPT-mask edit
4. 若恢复 GPT，先补齐遮罩生成与局部 edit 验证
5. 探索模型蒸馏，降低推理成本
