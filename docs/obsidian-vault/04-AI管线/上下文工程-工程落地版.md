# 上下文工程 - 工程落地版

> [!info] 姊妹篇
> 本文是**规格篇**，定义 schema、阈值、失败策略与实施计划。
> 核心理念与价值观边界见 [[上下文工程]]（理念篇）。
> **写代码以本文为准，遇到方向争议回理念篇找答案。**

> [!note]
> 这是对 [[上下文工程]] 的工程化改写版本。
> 保留其核心方向：`CameraSchema -> CreativeBrief`、参考图锚定、创意驱动质检。
> 但本稿更强调可执行性、失败回退、稳定性和维护成本。

---

## 一、结论先行

这套上下文工程方向是对的，但原方案偏重理念表达，还缺 4 个工程护栏：

1. 不能只去掉技术参数，还要保留少量弱控制字段，避免同套餐输出漂移过大。
2. 不能只说“参考图最重要”，还要定义参考图选择策略，否则身份锚定会互相污染。
3. 不能只让 VLM “对照创意审片”，还要把问题分成 hard fail 和 soft fail，否则会过度返工。
4. 不能默认 GPT 精修就是局部遮罩修复，必须区分局部修和全图修两种能力边界。

这份文档的目标不是“写得更美”，而是让后端能稳定接线，出问题时知道在哪一层兜底。

---

## 二、我们到底要解决什么问题

当前实现的真实问题不是“prompt 不够高级”，而是上下文职责混乱：

- Director 还在输出摄影参数，见 [director.py](/Users/elvis/Documents/codes/challenges/2026/Mar/AI-wedding-photos/src/backend/services/director.py#L40)。
- 生成 prompt 还是机械拼接 `scene + lighting + composition + lens`，见 [director.py](/Users/elvis/Documents/codes/challenges/2026/Mar/AI-wedding-photos/src/backend/services/director.py#L184)。
- variant 还是跨套餐通用描述，见 [generate.py](/Users/elvis/Documents/codes/challenges/2026/Mar/AI-wedding-photos/src/backend/routers/generate.py#L196)。
- VLM 质检还不知道这张图本来想表达什么，见 [vlm_checker.py](/Users/elvis/Documents/codes/challenges/2026/Mar/AI-wedding-photos/src/backend/services/vlm_checker.py#L43)。
- 参考图是“读到多少张算多少张”，没有主次选择，见 [generate.py](/Users/elvis/Documents/codes/challenges/2026/Mar/AI-wedding-photos/src/backend/routers/generate.py#L81)。

所以这次重构的核心，不是优化某一段 prompt，而是明确 5 件事：

1. 谁负责定义创意。
2. 谁负责锁定身份。
3. 谁负责控制变体。
4. 谁负责判断是否可交付。
5. 谁负责在失败时回退。

## 三、建议采用的上下文分层

仍然保留 5 层，但每层必须有输入、输出和失败策略。

### Layer 1: System Identity

职责：定义模型角色，不携带业务实例信息。

要求：

- 静态文本，放代码常量中。
- 不包含套餐名、场景名、用户偏好。
- 对 Nano、VLM、GPT Repair 分别维护。

失败策略：

- 这一层不依赖任何运行时数据，不允许动态生成。

### Layer 2: Creative Brief

职责：描述“这一套图要讲什么故事”。

要求：

- 每个套餐一份人工编写的基础 brief。
- Director 只允许编辑，不允许从零生成。
- 必须有默认值，Director 不可用时直接退回基础版本。

失败策略：

- `brief_base` 永远可用。
- `brief_edited` 可选。

### Layer 3: Identity Anchor

职责：明确“拍的是谁”，并约束参考图使用方式。

要求：

- 参考图必须有主图和辅图，不允��无限堆图。
- prompt 中必须明确说明：参考图是身份参考，不是风格参考。
- 质检时要复用同一身份锚定文本。

失败策略：

- 没有参考图时，管线仍可跑，但必须降级为“风格样片生成”，不能伪装成用户定制结果。

### Layer 4: Dynamic Slots

职责：承载本次请求的个性化内容。

包含：

- `makeup`
- `pairing`
- `user_preference`
- `wardrobe_override`

要求：

- 空值不渲染。
- 长文本必须被压缩成短句，不直接原样拼 prompt。
- 用户偏好要做类型区分，不要全塞进一句自然语言。

失败策略：

- 插槽解析失败时丢弃该插槽，不影响主干生成。

### Layer 5: Variant Plan

职责：定义同套餐下每张图的差异化目标。

要求：

- variant 必须和套餐绑定。
- variant 必须是“镜头意图 + 人物状态 + 画面关系”，不是纯 shot label。
- 每个 variant 应有自己的 avoid。

失败策略：

- 没有 variant 时，退回套餐默认 variant 列表。

## 四、CreativeBrief 不应是纯文案对象

原方案把 Creative Brief 写得很美，但工程上还不够稳。建议把它定义成“叙事字段 + 弱控制字段”的混合结构。

### 推荐最小 schema

```python
class CreativeBrief(BaseModel):
    package_id: str
    story: str
    visual_essence: str
    emotion: str
    aesthetic: str
    wardrobe_bride: str = ""
    wardrobe_groom: str = ""
    makeup_default: str = ""
    avoid_global: list[str] = Field(default_factory=list)
    shot_scale: str = "mixed"
    subject_arrangement: str = "couple"
    lighting_bias: str = ""
    pose_energy: str = "natural"
    variants: list["PromptVariant"] = Field(default_factory=list)
```

```python
class PromptVariant(BaseModel):
    id: str
    intent: str
    framing: str
    action: str
    emotion_focus: str = ""
    avoid_local: list[str] = Field(default_factory=list)
```

### 为什么要保留弱控制字段

如果完全删除 `lens / lighting / composition` 背后的控制意图，同一套餐的输出容易漂：

- 有时过近，婚纱细节丢失。
- 有时过远，脸部辨识度不足。
- 有时气氛对，但人物关系站位不稳定。

所以不建议回到旧 CameraSchema，但要保留低维约束：

- `shot_scale`: close / medium / wide / mixed
- `subject_arrangement`: face_to_face / side_by_side / embrace / walking
- `lighting_bias`: soft_warm / cool_diffused / high_contrast / neon_reflective
- `pose_energy`: still / tender / playful / dramatic

这些字段不是给人看的，是给 prompt assembler 用来稳住输出分布的。

## 五、参考图策略必须写成规则，不是理念

“参考图是最强上下文”这句话对，但还不够。工程上必须明确如何选图。

### 参考图选择规则

1. 主身份锚图：1 张
2. 辅助锚图：最多 2 张
3. 总参考图上限：3 张

### 主图选择优先级

1. 正脸清晰
2. 遮挡最少
3. 表情自然
4. 光线均匀
5. 美颜滤镜最弱

### 辅图选择优先级

1. 提供侧脸或不同角度
2. 提供半身或全身比例
3. 不与主图在妆容和滤镜上强冲突

### 明确禁止

- 不要把 6 张风格差异很大的自拍一起喂给模型。
- 不要混入妆造极重和极轻的两组脸作同一身份锚。
- 不要把情侣单人照和陌生样片混作参考图。

### 建议新增的后端职责

新增 `context/reference_selector.py`：

- 输入：用户上传图片列表
- 输出：`selected_refs`, `anchor_summary`
- 作用：在进入生图前先做一次参考图瘦身

## 六、Director 的职责重定义

Director 不再负责“凭空构思拍摄参数”，而是只做三件事：

1. 编辑 brief
2. 压缩偏好
3. 选择或改写 variant

### 是否调用 Director

| 情况 | 是否调用 | 原因 |
|------|----------|------|
| 用户无额外偏好 | 否 | 直接使用基础 brief |
| 只有妆造选择 | 否 | 机械插槽替换即可 |
| 用户输入自然语言偏好 | 是 | 需要把偏好压缩并融入 brief |
| 渠道不可用 | 否 | 直接 fallback 到基础 brief |

### Director 的输入

- `brief_base`
- `slot_payload`
- `reference_summary`
- `allowed_edit_scope`

### allowed_edit_scope 示例

```json
{
  "allow_edit": ["makeup_default", "variants", "wardrobe_bride", "wardrobe_groom"],
  "forbid_edit": ["story", "visual_essence", "package_id"]
}
```

### 这样做的目的

- 保住套餐核心叙事，不让 LLM 把“冰岛极光”修成“阿尔卑斯雪山”。
- 只允许 LLM 编辑边缘字段，降低跑偏概率。
- 为失败回退创造确定性。

## 七、Prompt Assembler 才是这次重构的核心

真正应该稳定的不是 Director 输出，而是最终 prompt 的组装器。

建议新增 `context/prompt_assembler.py`，职责只有一个：把结构化上下文渲染成稳定 prompt。

### Nano Prompt 组装顺序

1. System Identity
2. Identity Anchor
3. Creative Brief 主体
4. Dynamic Slots
5. Variant Intent
6. Weak Controls
7. Avoid Global + Avoid Local

### 示例骨架

```text
You are a world-class wedding photographer creating images that feel real,
emotional, and timeless.

The reference images show the real couple. Preserve their exact facial identity,
skin tone, body proportions, and overall likeness. These images are identity
references, not style references.

Story: ...
Visual essence: ...
Emotion: ...
Aesthetic: ...

Bride styling: ...
Groom styling: ...
Makeup: ...

This image should be a wide cinematic frame. They are walking away on black sand,
with a sense of private tenderness in a vast landscape.

Keep the framing readable, preserve facial recognizability, and maintain natural
body proportions.

Avoid: ...
```

### 关键原则

- 先讲身份，再讲创意。
- 先讲主叙事，再讲本张 intent。
- 约束语句不要多，但必须稳定。
- 所有套餐都必须走同一 assembler，不能每个套餐手写字符串。

## 八、VLM 质检不要只有一个总分

这是原方案最需要补实的地方。

如果只给一个 `score`，系统会把“六根手指”和“情绪不够空灵”都当成同类问题处理，这是错的。

### 建议拆成两层判定

#### Hard Fail

这类问题一旦出现，优先修或重生：

- 手部和肢体畸形
- 面部结构崩坏
- 明显身份不一致
- 衣物穿模
- 首饰嵌皮
- 大面积背景穿帮
- 关键人物模糊不可辨

#### Soft Fail

这类问题决定“是否够高级”，但不一定阻止交付：

- 情绪不足
- 互动不自然
- 氛围与套餐叙事不够一致
- 构图太平
- 妆造不够贴合
- 画面故事感不足

### 推荐 VLM 输出结构

```json
{
  "deliverable": true,
  "hard_fail": false,
  "identity_match": 0.88,
  "brief_alignment": 0.84,
  "aesthetic_score": 0.82,
  "issues": [
    {
      "description": "hands look fused near the waist",
      "category": "physical",
      "severity": 0.91,
      "blocking": true
    }
  ],
  "repair_mode": "local_fix",
  "repair_brief": [
    "separate the fingers and restore natural hand anatomy",
    "preserve the original composition and facial identity"
  ]
}
```

### 决策规则

1. `hard_fail = true` -> 修复或重生
2. `identity_match < threshold` -> 直接重生，不做情绪精修
3. `brief_alignment` 低但 `identity_match` 高 -> 允许 soft repair
4. 全部达标但不够惊艳 -> 可交付，不必为了美术追求无限返工

## 九、Repair 层必须区分局部修和全局修

当前实现里，修复层默认“能修”，但实际上能力边界没定义清楚。

建议把 repair 明确拆成两类：

### Local Fix

适用问题：

- 手指
- 衣物边缘
- 小范围光影错误
- 局部表情僵硬

要求：

- 优先使用局部能力或最小改动 edit
- prompt 必须包含 `preserve` 约束
- 修复后重新走 VLM

### Global Regenerate

适用问题：

- 身份不像
- 构图完全不对
- 情绪方向错误
- 套餐氛围整体跑偏

要求：

- 不要在错误底图上反复修补
- 直接重生更便宜也更稳定

### 一条重要原则

不要把“情绪不对”默认为 edit 问题。很多时候它不是修出来的，是重生出来的。

## 十、建议的目录结构

```text
src/backend/
  context/
    __init__.py
    briefs.py
    reference_selector.py
    prompt_assembler.py
    variant_planner.py
```

### 各模块职责

- `briefs.py`: 套餐基础 brief 模板库
- `reference_selector.py`: 主图/辅图筛选
- `prompt_assembler.py`: 最终 prompt 组装
- `variant_planner.py`: 套餐绑定的 variant 定义
- `slot_renderer.py`: 妆造、偏好、pairing 渲染
- `thresholds.py`: VLM 判定阈值与交付策略

## 十一、建议的实施顺序

### Phase 1: 最小闭环

先只做两套：`iceland`、`cyberpunk`

交付物：

- `CreativeBrief` schema
- 2 套基础 briefs
- reference selector
- prompt assembler
- variant planner

目标：

- 不改前端接口
- 先让两套套餐稳定跑通

### Phase 2: 质检升级

交付物：

- VLM 新 schema
- hard/soft fail 判定
- repair_mode 决策

目标：

- 让“修”和“重生”分流明确

### Phase 3: Director 编辑模式

交付物：

- Director 只编辑允许字段
- 无偏好时不调 Director
- fallback 完整接入

目标：

- 把 LLM 从关键路径“必须成功”降为“增强器”

### Phase 4: 扩到全部套餐

交付物：

- 剩余 8 套 brief
- 套餐特定 variants
- 联调迭代记录沉淀到知识库

## 十二、这版方案与原稿的核心分歧

我认可原稿的方向，但这里有 3 个明确分歧：

1. 原稿强调“少写技术参数”，我同意，但我主张保留弱控制字段。
2. 原稿强调“参考图最强”，我同意，但我主张必须先做参考图筛选，而不是直接多图全喂。
3. 原稿强调“对照创意审片”，我同意，但我主张必须拆出 hard fail，否则质检会变成高成本审美挑刺。

换句话说：

- 原稿更像“认知哲学”
- 本稿更像“可上线规格”

两者不是互斥关系。最好的结果不是二选一，而是：

原稿做价值观边界，本稿做执行约束。

---

## 十三、最终建议

如果只能给一句建议，那就是：

**不要把这次重构理解为“把 prompt 写得更像摄影师”，而要理解为“把上下文从松散文本升级成有层次、有阈值、有回退的系统”。**

这才是它能长期维护、能稳定出片、能持续调优的前提。
