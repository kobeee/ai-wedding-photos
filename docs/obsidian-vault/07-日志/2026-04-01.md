## VPS 线上验证再补记：根因确认并修复

这轮继续往下排后，真正的主阻塞已经确认不是模型质量本身，而是生成管线里的一个轮次语义 bug：

### 根因

- VPS 当前运行时配置里为了控成本，使用了：
  - `MAX_FIX_ROUNDS=1`
- 但代码原先把 `MAX_FIX_ROUNDS` 直接当成 `质检总轮数`
- 结果就是：
  - 第一轮生成后进入第一次质检
  - 这时 `fix_round=0`
  - 同时又被判定成“最后一轮”
  - 只要 `hard_fail` 或低分 soft fail，就直接 `reject`
  - 根本不会进入任何一次真实修复

换句话说：

- 配置里写的是“允许 1 轮修复”
- 实际代码执行成了“允许 0 次修复”

这正好解释了上一轮线上日志里的现象：

- `quality_check`
- 紧接着 `Photo 1 decision: reject (Hard fail remained after final round)`

### 本轮修复

- 调整生成管线轮次语义：
  - `MAX_FIX_ROUNDS` 现在表示“初次生成后允许的修复次数”
  - 质检总轮数改为 `1 + MAX_FIX_ROUNDS`

## Upload 413 再兜底

这轮继续回看“点击 AI 试妆时报上传内容过大”的链路，结论是：

- 现有 `96M nginx + 前端自动压缩到约 3MB/张` 已经显著降低了风险
- 但只要前端仍把 `5` 张图一次性 `POST /api/upload`
- 那么代理层总请求体上限依旧是一个单点

### 本轮补强策略

- 前端保留批量上传主路径
- 如果批量上传返回 `413`
- 自动降级为 `5` 次逐张上传，而不是直接报错中断
- 单张降级上传仍然携带 `role + slot`

后端同步做了一个兼容调整：

- 只有在一次请求里拿到完整必传坑位集合时，才触发整包上传校验
- 单张带 `slot` 上传时只负责落库和写 metadata，不再因为“缺少其他坑位”直接报错

### 这次验证

- 后端回归：
  - `src/backend/tests/test_upload_validation.py` 通过
  - 新增覆盖：单张 `slot` 上传不会触发整包校验
- 前端回归：
  - `src/frontend` 执行 `npm run build` 通过

### 当前结论

- 即使未来再次遇到代理层 `413`
- 上传页也会自动拆成逐张请求继续完成流程
- 这次修补的目标不是继续“赌上限够大”，而是把总请求体限制从主链路里移掉

## AI 试妆超时降级修复

### 问题

用户进入 AI 试妆页面后，等了很久，最终看到提示”AI 试妆当前响应较慢，已切换为样片预览，不影响正式生成”。

### 排查

SSH 到 VPS 查看后端日志 `docker logs wedding-photos-backend-1`：

```
13:52:32 [INFO] routers.makeup: Using reference image: ...
13:53:04 [INFO] httpx: POST ... generateContent “HTTP/1.1 200 OK”  # 第1张 ~32s
13:53:37 [INFO] httpx: POST ... generateContent “HTTP/1.1 200 OK”  # 第2张 ~33s
13:54:01 [INFO] httpx: POST ... generateContent “HTTP/1.1 200 OK”  # 第3张 ~24s
13:54:01 172.25.0.4 - “POST /api/makeup/generate” 200 OK           # 总计 ~89s
```

关键发现：

1. 每次 Nano Banana 调用耗时 20-32 秒（图生图正常水平）
2. 后端 3 种妆造风格是**串行**调用的：`for style in ALL_STYLES: await _render_style(style)`
3. 3 次串行 = 60-90 秒
4. 前端 `requestMakeupPreview()` 的 `AbortController` 超时只设了 **20 秒**
5. 20 秒连第一张图都等不完 → 必然 abort → 降级到样片预览

**不是 laozhang 并发问题，是超时设置 + 串行调用的双重问题。**

### 修复

| 位置 | 之前 | 之后 |
|------|------|------|
| `src/backend/routers/makeup.py` | `for style in ALL_STYLES: await _render_style(style)` 串行 | `asyncio.gather(*(_render_style(s) for s in ALL_STYLES))` 并行 |
| `src/frontend/src/pages/Makeup.tsx` | `setTimeout(() => controller.abort(), 20000)` | `setTimeout(() => controller.abort(), 90000)` |

预期效果：
- 后端耗时从 ~90s 降到 ~30s（3 张并行，取最慢的一张）
- 前端给 90s 余量，足够等并行完成

### 部署

- 通过管道方式（`cat file | ssh ... “cat > path”`）上传两个改动文件
- `docker compose down && docker compose up -d` 重启
- 期间清理了旧的 `wedding-photos-*` 残留容器（和新的 `docker-*` 容器冲突占端口）
- `curl https://wedding.escapemobius.cc/api/health` 返回 200 确认服务正常

### 额外发现

VPS SSH 连接不稳定（频繁 `Connection closed` 和 `Connection timed out`），rsync 大批量同步容易断。小文件用管道方式传输更可靠。
