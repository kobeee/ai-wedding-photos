# 2026-03-24

## 结论先行

- 这次真正的问题不是“代码没修”，而是 **VPS 上同时存在两套 Compose project**
- 实际对外流量走的是 `wedding-photos-*`
- 一开始误更新、误验证了 `docker-*` 这套临时栈，所以看起来像“代码已改但线上行为没变”
- 当前生产栈已经切回正确 project，并完成重新部署与验证

## 本次定位出的真实问题

### 1. 部署目标打错 project name

- 同一个 `docker/docker-compose.yml`
- 用默认 project name 会起出 `docker-*`
- 线上真实运行的却是 `wedding-photos-*`
- 结果：
  - 本地以为已经发布成功
  - 实际 `3080` 仍在服务旧后端

### 2. 交付口径不符合产品目标

- 旧行为会出现：
  - `quality_score = 0.50`
  - 任务仍然 `completed`
  - 甚至返回结果图
- 这不符合当前产品定位
- 对高定婚纱产品来说，低分图和未审图都不应“勉强交付”

### 3. Director 在线编辑存在截断 JSON 风险

- `Director` 调 `chat/completions` 时，模型偶发返回半截 JSON
- 旧实现直接 `json.loads(raw)`，一旦失败就抛异常并整体 fallback 到基础 brief
- 这不会炸主流程，但会丢失用户偏好

### 4. VLM 质检存在外部超时风险

- 当前 `laozhang.ai` 的视觉聊天在生产上仍可能 `ReadTimeout`
- 这类错误现在不再静默放行，而是明确���止交付

## 本次代码修正

### 交付策略

- `thresholds.py`
  - 新增 `RepairMode.reject`
  - 最后一轮 `identity low` / `hard fail` 不再强行交付
  - 新增 `MIN_DELIVERY_FLOOR = 0.72`
  - 新增 `meets_delivery_floor()` 最终交付闸门

- `generate.py`
  - 被 `reject` 的图不再落盘
  - 所有结果在保存前再过一次最终交付分闸门
  - 没有可交付结果时，任务直接 `failed`

### VLM 失败语义

- `vlm_checker.py`
  - fallback report 改为：
    - `passed = false`
    - `inspection_unavailable = true`
    - `score = 0.0`
  - `generate` 遇到 `inspection_unavailable` 时直接失败
  - 明确阻止“质检不可用但仍交付”

### 身份锚定

- `upload.py`
  - 上传新增 `roles` 字段
- `storage.py`
  - 为上传文件写入 sidecar metadata
- `reference_selector.py`
  - 优先保留 bride / groom 双人身份锚
  - 暴露 `has_couple_identity`

### Director 鲁棒性

- `director.py`
  - 不再只做一次 `json.loads`
  - 解析失败后会二次请求更强约束的 JSON
  - 仍失败时尝试从 code fence、噪音文本、截断 key/value 中做兜底提取
  - 最后才回退到基础 brief

## VPS 真实验证结果

### 生产栈确认

- 当前有效 Compose project：`wedding-photos`
- 当前误起的 `docker-*` 栈已清理
- 当前公网对应前端容器仍是：
  - `wedding-photos-frontend-1`

### 页面与接口

- `/` → `200`
- `/upload` → `200`
- `/makeup` → `200`
- `/package` → `200`
- `/waiting` → `200`
- `/review` → `200`
- `/api/health` → `200`

### 生产环境全链路

基于最新生产栈重新跑了：

1. `POST /api/upload`
2. `POST /api/generate`
3. 轮询 `GET /api/generate/{task_id}/status`
4. 查询 `GET /api/generate/{task_id}/result`

最终结果：

- 任务未再出现“0.50 低分也 completed”的旧行为
- 在 VLM 超时场景下，任务终态为：
  - `status = failed`
  - `message = 质量检查服务暂不可用，已停止交付以避免输出未审图像`
- `result` 接口也不再返回伪成功结果，而是返回同样的失败原因

这说明新的系统行为已经符合产品目标：

- 宁可失败
- 不交付低质图
- 不交付未审图

## 当前剩余问题

### 已解决的内部问题

- Compose project 误判
- 低分图误交付
- 质检不可用静默放行
- Director 解析异常直接打断偏好编辑

### 尚存的外部风险

- `laozhang.ai` 的 VLM 通道仍可能超时
- 当 VLM 超时发生时，当前策略是安全失败，而不是降级交付

这不是当前代码 bug，而是外部能力稳定性问题。

## 当前可执行结论

- 生产后端的“交付语义”已改正
- 现在的系统更接近高定产品应有的行为：
  - 失败是可接受的
  - 错交付不可接受
- 后续若要把成功率拉高，优先级不在 prompt，而在：
  - 稳定的 VLM 质检通道
  - 更可靠的外部模型响应质量
