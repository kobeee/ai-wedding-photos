# 2026-03-25

## Nano-only 修复链路落地

- 背景：主链路里 GPT-Image-1.5 只用于 emotional repair，但实际调用没有传 mask，VLM 质检也不产出可直接用于 edit 的区域信息，导致这条链路更接近“全图盲修”而不是预期中的局部精修。
- 决策：生产默认切到 **Nano Banana 全修复**。
- 保留：`GPTImageService`、模型配置和开关仍保留，便于后续实验恢复。

## 本次代码改动

- `config.py`
  - 新增 `enable_gpt_image_repairs = false`
- `generate.py`
  - physical / emotional 两类问题默认统一走 Nano 修复
  - emotional 分支仅在显式打开开关时才尝试 GPT，失败后回落 Nano
- `nano_banana.py`
  - 新增 `repair_with_references()`，修复时同时喂当前成片和身份参考图
- `prompt_assembler.py`
  - 新增 `assemble_nano_repair_prompt()`，把原始创意意图、修复 focus、preserve 约束和身份参考角色一起喂给 Nano
- `acp/agents/photographer.py`
  - Agent 修复环节与 HTTP 主链路对齐，统一走 Nano repair prompt

## 这样做的原因

- 当前没有 mask，就不该把 GPT edit 当成“局部精修”来做成本和效果判断
- Nano 已经是主渲染模型，统一修复可以减少跨模型风格漂移
- 多图修复把“当前成片 + 身份参考”同时送入模型，更符合婚纱照对 ID 稳定性的要求
- 在没有实测数据前，减少调用链路更符合 [[战略决策稿]] 里的成本口径

## 后续恢复 GPT 的前提

1. VLM 或额外模块能稳定给出 face / mouth / eye 区域定位或 mask。
2. 建立一组固定测试样本，对比 Nano-only repair 与 GPT-mask edit 的：
   - 修复成功率
   - 单轮延迟
   - 单张成本
   - ID 漂移率
3. 只有当效果收益明确超过成本和复杂度，才恢复双模型修复默认值。
