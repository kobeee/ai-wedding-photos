# 2026-03-21

## 模型基线修正

- 用户已明确当前系统的目标三模型栈：
  - `gemini-3.1-pro-preview`
  - `gemini-3-pro-image-preview`
  - `gpt-image-1.5`
- 因此前一轮把 `gpt-4o` 当成 Director / VLM 的默认模型是假设错误
- 后端默认配置已改为：
  - `director_model = gemini-3.1-pro-preview`
  - `vlm_model = gemini-3.1-pro-preview`

## 结论先行

- 当前 `LAOZHANG_API_KEY` 可用的是真实生图链路
- 当前 `LAOZHANG_API_KEY` **不可用** 的是 `chat/completions` 路由
- 因此问题不是“laozhang 偶发不稳定”，而是“当前 key / group 对聊天模型没有可用渠道”

## 直接探针结果

### `/v1/chat/completions`

对以下模型做了最小文本探针，均返回 `503`，错误文案一致：

- `gpt-4o`
- `gpt-4o-mini`
- `gpt-4.1-mini`
- `gpt-4.1`
- `chatgpt-4o-latest`
- `claude-3-5-haiku-latest`
- `claude-3-5-sonnet-latest`
- `claude-3-7-sonnet-latest`

典型错误：

```json
{
  "error": {
    "message": "当前分组 default 下对于模型 gpt-4o 计费模式 [按次计费] 无可用渠道",
    "type": "shell_api_error"
  }
}
```

### `/v1/models`

- `models` 接口能列出大量模型 ID
- 但“能列出模型”不代表“当前 key / default group 真的有可用渠道”
- 所以这条接口只能证明平台声明支持，不足以证明当前账号实配可用

### Gemini 图像端点

- `POST /v1beta/models/gemini-3-pro-image-preview:generateContent` 继续稳定返回 `200`
- 说明这把 key 的问题不是“完全不可用”，而是能力面不对称：
  - 图像生成通
  - 聊天 / 视觉聊天不通

## 后端代码调整

### 目标

避免每次请求都重复打不通的 `chat/completions`，造成：

- 无意义等待
- 噪音日志
- 误导性的 `quality_score=0.00`

### 处理方式

- `DirectorService`
  - 首次收到“无可用渠道”后，缓存 `channel unavailable`
  - 后续请求直接走本地 fallback，不再访问 `chat/completions`

- `VLMCheckerService`
  - 首次收到“无可用渠道”后，缓存 `channel unavailable`
  - 后续请求直接返回本地 fallback report
  - fallback score 从原来的 `0.00` 调整为 `quality_acceptable (0.85)`

## 二次生成验证

backend 重启后，连续执行两次低成本 `generate`：

### 第一次

- Director 命中一次 `503`
- VLM 命中一次 `503`
- 两个服务均记录 “future requests will use local fallback”
- 任务成功完成，`quality_score = 0.85`

### 第二次

- 不再出现新的 `chat/completions` 请求
- 只保留 Gemini 生图请求
- 任务再次成功完成，`quality_score = 0.85`

## 当前真实状态

- “上传 → 鉴权 → 生图 → 结果交付” 已测通
- “妆造 → 产出三张结果图” 已测通
- “Director / VLM 真实在线模型推理” 在当前 key 下不可测
- 后端现已进入“探测一次后自动降级”的稳定模式，不会每次重复撞聊天通道

## 后续动作建议

1. 若要真正恢复完整双模型闭环，需要 laozhang 侧给当前 key 开通可用的 `chat/completions` 渠道
2. 若短期无法开通，可考虑把 Director / VLM 改成完全本地 fallback 驱动的产品模式
3. 前端后续接真实 API 时，应知晓：
   - 当前 `quality_score=0.85` 可能来自 fallback
   - 并不代表 VLM 真正完成了视觉质检

## 晚间修正（2026-03-21）

### 本次联调的真实结论

- 后端默认模型已统一为顶配组合：
  - `director_model = gemini-3.1-pro-preview`
  - `vlm_model = gemini-3.1-pro-preview`
  - `gpt_image_model = gpt-image-1.5`
  - `nano_banana_model = gemini-3-pro-image-preview`
- `upload -> session_token -> 鉴权` 链路可用
- `Director` 本轮已能推进到生成阶段前，不再停在早期 JSON 解析失败
- 当前真正阻断全链路的点仍是 `gemini-3-pro-image-preview`

### 新发现的运维问题

- `docker/docker-compose.yml` 中原先使用了映射式 `env_file`
- VPS 上的 Docker Compose 实际没有把该写法解析进最终配置
- 结果是 `backend` / `acp` 容器启动时并未读取 `.env.runtime`
- 修复方式：
  - 改回标准写法：
    - `env_file:`
    - `  - ../.env.runtime`

### 本次事故复盘

- 一次 `rsync --delete` 把 VPS 上私有维护的 `/opt/apps/wedding-photos/.env.runtime` 删除了
- 该文件不在仓库内，这是符合安全原则的，但也意味着同步脚本必须显式保护它
- 已恢复该文件权限为仅 root 可读写，并重新注入：
  - `LAOZHANG_API_KEY`
  - `PHOTOS_PER_PACKAGE=1`
  - `MAX_FIX_ROUNDS=1`

### 重新验证结果

- 容器内健康检查：
  - `api_configured = true`
  - `gpt_image_model = gpt-image-1.5`
- 最小链路再次验证：
  - `POST /api/upload` 成功
  - `POST /api/makeup/generate` 失败，返回 `All makeup styles generation failed`
  - `POST /api/generate` 能创建任务
  - 任务状态会推进到 `progress=15`，即“正在生成第 1/1 张”
  - 随后失败，说明阻塞点已经落在图像生成而不是 Director 初始化

### 当前可执行结论

- `gemini-3.1-pro-preview` 已是后端在线推理模型
- `gpt-image-1.5` 已是后端在线修图模型配置
- 但只要当前 key / 分组下 `gemini-3-pro-image-preview` 无可用渠道：
  - `makeup`
  - `generate`
  这两条核心图像链路都无法真正打通

### 后续约束

- VPS 私有密钥文件不得再通过仓库同步覆盖
- 后续部署脚本必须避开：
  - `.env.runtime`
  - 其他运行态私有文件
- 在未明确需要生图时，继续避免额外调用付费图像接口

### 当晚最终落地修正

- 当前 laozhang 需按模型能力拆分两把 key：
  - `LAOZHANG_NANO_API_KEY`
    - 仅供 `gemini-3-pro-image-preview`
  - `LAOZHANG_API_KEY`
    - 供 `gemini-3.1-pro-preview`
    - 供 `gpt-image-1.5`
- 后端已改为双 key 路由，并保留单 key 向后兼容 fallback
- `NanoBananaService` 调用 `v1beta/models/...:generateContent` 时，认证头必须使用：
  - `x-goog-api-key`
- 不能再用：
  - `Authorization: Bearer ...`

### 线上最终验证结果

- `POST /api/makeup/generate`
  - 成功
  - 实际产出 3 张妆造图
- `POST /api/generate`
  - 成功创建任务
  - 最终 `status = completed`
  - 最终结果图 1 张
- 当前残余问题不是链路不通，而是模型输出稳定性：
  - `Director` 在线返回仍可能给出截断 JSON
  - 当前实现会 fallback 到本地 schema，任务仍能完成
  - `VLM Checker` 在线返回也可能给出截断 JSON
  - 本轮因此落到了 `quality_score = 0.50`

### 部署层新坑

- VPS 上同时存在两套 compose project：
  - 真正在接流量的是 `wedding-photos-*`
  - 误操作会起出一套 `docker-*`
- 若使用：
  - `docker compose -f docker/docker-compose.yml ...`
  默认 project 名会变成 `docker`
- 这会导致“代码已重建，但 3080 仍走旧容器”
- 正确做法：
  - `docker compose -p wedding-photos -f docker/docker-compose.yml ...`
