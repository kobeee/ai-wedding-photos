# 2026-03-20

## AI 联调原则落地

- `LAOZHANG_API_KEY` 只保存到 VPS 运行时环境文件：`/opt/apps/wedding-photos/.env.runtime`
- 密钥不写入仓库、不写入 Obsidian、不写入源码常量
- 当前线上 `backend` 与 `acp` 通过 `docker/docker-compose.yml` 的 `env_file` 在运行时注入该密钥
- 联调期间默认启用低成本阈值：
  - `PHOTOS_PER_PACKAGE=1`
  - `MAX_FIX_ROUNDS=1`
- 只有在“明确要验证真实生图链路”时才调用 laozhang.ai
- 普通测试图、占位图、流程前置素材优先尝试 Pencil 自带的图片能力，尽量不要消耗付费额度

> [!warning]
> 不要在共享终端或日志里直接执行会展开敏感环境变量的命令，例如 `docker compose config`。该命令会把 `env_file` 中的敏感值展开到输出。

## Pencil 图片能力结论

### 当前状态

- 本机本轮未能直接实测 Pencil 生图，因为 MCP 连接 Pencil app 时返回 WebSocket 未连接

### 基于工具定义的结论

- Pencil 支持两类图片来源：
  - `G(nodeId, "ai", prompt)`：AI 生成图片
  - `G(nodeId, "stock", prompt)`：检索 stock 图
- 图片不是独立节点，而是作为 `frame/rectangle` 的 fill 应用
- 若要把图片拿到项目里使用，需要先放进 `.pen` 画布，再导出节点
- 这更适合：
  - 测试自拍占位图
  - 普通参考图
  - 设计稿里的气氛图和示意图
- 不适合直接依赖它做：
  - 高一致性的人脸身份保真
  - 后端生产链路验证
  - 需要稳定复现的付费结果基线

> [!info]
> 一旦 Pencil app 连接恢复，可补做一次实测，重点验证普通自拍、双人合照、stock 搜图三个场景。

## 本次真实链路验证

### 1. VPS 密钥注入

- 已在 VPS 写入 `/opt/apps/wedding-photos/.env.runtime`
- 已确认公网 `https://wedding.escapemobius.cc/api/health` 返回 `api_configured: true`

### 2. 核心婚纱生成链路

- 使用现有测试图走 `/api/upload`
- 上传成功后确认拿到：
  - `user_id`
  - `session_token`
- 未带 `Authorization` 调用 `/api/generate` 返回 `401`
- 带 `Authorization: Bearer <session_token>` 后成功创建任务
- 任务最终完成并产出 1 张结果图
- 本地代理与公网结果 URL 均返回 `200`

### 3. 妆造链路

- `/api/makeup/generate` 的后端实际成功生成了 3 张妆造图
- 首次客户端请求拿到的是 `504 Gateway Time-out`
- 根因不是模型失败，而是前端 Nginx 代理超时过短
- 补齐代理超时后，妆造产出图的本地代理与公网 URL 均返回 `200`

## 本次发现的真实阻塞点

### 阻塞 1：上传响应模型漏了 `session_token`

- 现象：`upload.py` 返回了 token，但 `UploadResponse` schema 未声明该字段
- 影响：前端即使上传成功，也无法获得后续鉴权所需 token
- 处理：在 `src/backend/models/schemas.py` 中补回 `session_token`

### 阻塞 2：公网上传被 Nginx 413 拦截

- 现象：上传 1.6MB PNG 时收到 `413 Request Entity Too Large`
- 根因：前端 Nginx 未配置 `client_max_body_size`
- 处理：在 `docker/nginx-frontend.conf` 中设置 `client_max_body_size 20M`

### 阻塞 3：妆造接口被代理 504 截断

- 现象：`/api/makeup/generate` 后端仍在生图，但前端已返回 `504`
- 根因：该接口同步串行生成 3 张图，耗时超过代理默认超时
- 处理：在 `docker/nginx-frontend.conf` 中补充：
  - `proxy_connect_timeout 300s`
  - `proxy_send_timeout 300s`
  - `proxy_read_timeout 300s`
  - `send_timeout 300s`

## 上游风险观察

- laozhang.ai 的 `POST /v1/chat/completions` 本轮多次返回 `503`
- 影响范围：
  - `Director` LLM 自动降级到 fallback prompt
  - `VLMChecker` 质检降级为 fallback 结果
- 当前生成任务虽然完成，但综合质量分数被记为 `0.00`

> [!warning]
> 这说明“生图链路可通”不等于“质检链路稳定”。后续应补：
> 1. `chat/completions` 的重试与退避
> 2. VLM fallback 分数语义校正
> 3. 妆造接口改后台任务化，避免同步等待过久

## 当前结论

- 线上真实付费链路已经打通
- 密钥管理方式已符合“不入库、只存 VPS 运行时环境”的要求
- 核心问题已不是“能不能生成”，而是：
  - 上游 chat/completions 稳定性
  - 妆造接口的同步耗时设计
  - 前端尚未完整接入真实 API 流程
