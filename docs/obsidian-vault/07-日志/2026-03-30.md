## 2026-03-30 晚间补充：上传页固定坑位与 AI 入库校验

### 背景

- 现有上传页只告诉用户“传几张大概的日常照”
- 对用户来说：
  - 不知道到底该传哪几张
  - 不知道合照是否必要
  - 不知道系统会不会把不同人的图混到一起
- 线上还出现了：
  - `{"error":"请求体积过大，请减少内容后重试 ..."}`

### 本轮处理

#### 前端

- Upload 页面改为固定坑位：
  - `groom_portrait`
  - `groom_full`
  - `bride_portrait`
  - `bride_full`
  - `couple_full`（可选）
- 每个坑位只收 1 张
- 页面直接给“图片示意 + 取景框”：
  - 用中性构图示意，不再绑定任何族裔标签
  - 明确“只看取景范围，不要求同款穿搭”
- 浏览器侧上传前压缩：
  - 最大长边约 `2200px`
  - 自动转 JPEG
  - 单张目标约 `2.6MB`

#### 后端

- `/api/upload` 新增 `slots`
- sidecar metadata 新增：
  - `slot`
  - `validation`
- 新增 AI 上传校验：
  - 看坑位是否拍对
  - 看同角色两张是否像同一个人
  - 若有合照，再看是否像同一对人
- 校验失败直接阻断上传
- 参考图筛选与试妆逻辑同步读取新 metadata

#### Payload 收敛

- 上传落盘前统一压缩
- Nano Banana 参考图入参也统一缩图
- 目标不是只挡单张超限，而是减少整包和下游模型 payload 过大概率

### 本地验证

- `src/frontend` 已执行 `npm run build` 通过
- 后端测试已用项目 `.venv` 执行通过：
  - `test_upload_validation.py`

- 已本地截图复查桌面与移动端上传页，视觉风格保持深色奢华调性

## 2026-03-30 深夜补充：参考图预算裁剪与“抽象图”回收

### 触发背景

- VPS 上继续出现“人物抽象、不像本人”的结果
- 同时上游偶发返回：
  - `{"error":"请求体积过大，请减少内容后重试 ..."}`
- 说明问题不只是“有没有参考图”，而是：
  - 上传校验阶段一次送太多图
  - 生成阶段参考图过多、过大，既可能触发 payload 超限，也可能稀释身份锚图

### 本轮处理

#### 上���校验侧

- AI 校验改为按固定优先级裁剪：
  - 先保留 `groom_portrait / bride_portrait / groom_full / bride_full`
  - `couple_full` 只在预算允许时才进入 AI 校验
- 每张校验图继续缩到小图，但新增更强的字节预算：
  - 单张约 `180KB` 上限
  - 总量约 `720KB` 上限
- 图片预处理增加 `EXIF` 转正，避免手机竖图方向错乱影响校验

#### 生成侧

- Nano Banana 参考图改为“压缩后再按总预算裁剪”，不再无脑全塞：
  - 单张约 `220KB` 上限
  - 总量约 `880KB` 上限
- 检测到上游 `请求体积过大` 时，后端会返回更明确的用户提示，而不是盲目重试 3 次
- `repair_with_references` 也同步走同一套压缩与裁剪逻辑

#### 参考图选择策略

- 有 `couple_full` 时：
  - 只保留 `couple_full + groom_portrait + bride_portrait`
  - 不再继续追加两张全身图
- 没有合照时：
  - 优先保留新郎/新娘正脸图
  - 再补各自全身图
- 核心目标：
  - 先稳住身份
  - 再补身形
  - 避免参考图数量过多导致模型“平均脸化”

### 本地验证

- 使用项目 `.venv` 通过：
  - `src/backend/tests/test_upload_validation.py`
  - `src/backend/tests/test_reference_selector.py`
  - `src/backend/tests/test_order_closure.py`

### 当前判断

- 这轮不是继续堆 prompt，而是把“参考图预算”做成硬约束
- 如果线上仍有抽象图，后续更应该优先看：
  - 选入的 3 张参考图是否真的对
  - 是否还需要把生成阶段再收紧到 `2-3` 张
  - 是否需要对 couple 场景单独做更强的身份提示词

### 深夜最后补记：用户线上反馈与当前未闭合问题

- 我又尝试把上传页左侧的“参考图示意”从 CSS 抽象块替换成轻量 SVG 引导图
- 本地前端 `build` 通过，也重新发布了 `frontend`
- 但用户在 VPS 实际页面截图里，看到的仍然是旧的抽象占位块，而不是新的示意图

当前必须以用户线上截图为准，结论如下：

- “上传页参考图示意已修好” 这个结论不成立
- 当前问题没有闭合，至少还存在以下一种情况：
  - 发布的不是用户最终看到的前端资源
  - CDN / 浏览器缓存命中了旧资源

- 后端参考图预算裁剪已上线并验证过
- 但上传页参考图示意的前端替换，用户侧验收失败
- 后续若继续处理，必须直接围绕“用户实际看到的线上 DOM / 静态资源”排查，不能再按本地截图或 agent 自己抓图判定完成

## 2026-03-30 深夜再补记：问题重新定性并修正上传原图策略

这次根据用户追问，我把上传页问题重新拆成了两个层面：

- 第一层：VPS 当前页面的示意图并不是“缓存残影”，而是打包产物里真的还在引用 `upload-guides/*.svg`
- 第二层：浏览器上传前压缩虽然能缓解 body size，但它直接牺牲了用户原图，这不是正确的长期方案

### 重新核对后的事实

- SSH 到 VPS 后确认：
  - `src/frontend/src/pages/Upload.tsx`
  - `src/frontend/dist/assets/*.js`
  都仍然引用 `upload-guides/*.svg`
- 说明用户看到抽象图，不是错觉，也不是纯缓存问题

### 本轮修正

#### 上传页视觉

- 把 `upload-guides/*.svg` 线稿替换成项目内已有真人样片裁切版 `*.webp`
- 保留取景框和标签，只替换底图素材
- 上传页文案整体收短，去掉重复解释

#### 上传原图策略

- 前端不再把用户文件转 JPEG 再上传
- 后端保存上传文件时也不再做二次有损压缩
- 现在的分层改为：
  - 上传与落盘：保留原文件
  - AI 校验前：生成预算化小图
  - Nano Banana 参考图入参前：再生成预算化小图

### 本地验证

- `src/frontend`:
  - `npm run build` 通过
- `.venv`:
  - `python -m unittest src/backend/tests/test_upload_validation.py src/backend/tests/test_reference_selector.py` 通过
- 新增断言：
  - `/api/upload` 会保留原始 `PNG` 扩展名
  - 落盘字节与原始上传内容一致

### 当前判断

- “上传前先压一轮，后面就更稳” 这个思路只适合临时止血，不适合长期产品链路
- 更合理的策略已经明确：
  - 原图是资产，应该短期保存
  - 压缩只应该发生在外发给模型的派生副本上
