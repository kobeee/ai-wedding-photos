# 2026-03-27

## VLM 不可用时恢复可交付降级

- 线上真实验证再次确认：当前 key 仍然没有 `chat/completions` 可用渠道。
- 这会导致 `VLMCheckerService` 返回 `inspection_unavailable = true`。
- 3 月 24 日后的严格策略会让 `generate` 直接失败，结果是整个线上生成链路无法交付。

## 本次修正

- 保留 `vlm_checker.py` 的严格语义：
  - `inspection_unavailable = true`
  - `score = 0.0`
  - 不伪造“质检真的成功”
- 把降级决策收回到 `generate.py`
  - 新增配置 `allow_degraded_delivery_on_vlm_unavailable = true`
  - 当 VLM 不可用时，HTTP 主流水线不再整单失败
  - 当前照片按 `quality_acceptable` 降级交付，并在任务消息中显式标注“质检降级”

## 原因

- ACP agent 之前本来就是遇到 `inspection_unavailable` 后直接 break，再交付图片；HTTP 路径反而更严格，造成两套行为不一致。
- 当前生产 key 的真实能力面是：
  - Nano 生图可用
  - chat/completions 不可用
- 如果继续坚持“VLM 不通就整单失败”，那当前生产环境根本无法完成完整生成闭环。

## 结论

- 这次修正不是否定质检，而是承认当前上游能力现状，把“降级交付”的责任放到配置层和任务消息层，而不是假装质检正常。
- 等 laozhang 为当前 key 恢复可用 `chat/completions` 渠道后，可以把该开关关回严格模式。

## 晚间补充：VLM timeout 根因修正

- 用户提供 laozhang 消费日志后，进一步确认：
  - `gemini-3.1-pro-preview` 调用并非彻底不可用
  - 历史同类请求耗时最高已到 `68s`
  - 当前代码里 `vlm_checker.py` 的本地 timeout 只有 `60s`
- 结论：
  - VLM 的问题更接近“重 payload + 尾延迟超过本地 timeout”
  - 不是 Nano-only 改动导致，也不等于上游完全没渠道

- 修正：
  - `vlm_timeout_seconds` 提高到 `120`
  - `vlm_max_tokens` 提高到 `2048`
  - VLM 质检前先把图片压到较小长边，并统一转成 JPEG
  - 增加 VLM 请求前后的尺寸与耗时日志，方便后续继续定性
  - 请求增加 `response_format = json_object`，降低 JSON 被截断的概率

## 补充收口：正式交付改为 4K 输出

### 设计决策

- “婚纱大片要 4K” 这个产品判断成立，但不应该把首轮 Nano 生图直接强制到 4K。
- 当前采用的是更稳的两段式：
  - Nano / VLM 主链路继续走当前稳定分辨率
  - 只有通过质检、准备落盘交付时，才把正式成片升档到 `4096` 长边
- 这样能同时满足：
  - 用户下载的是 4K 大图
  - 不重新引入首轮生成耗时/失败率飙升

### 代码实现

- `src/backend/config.py`
  - 新增 `delivery_long_edge = 4096`
- `src/backend/services/delivery_image.py`
  - 新增正式交付升档服务
  - 逻辑：
    - 保持原图比例
    - 长边统一拉到 `4096`
    - 使用 `LANCZOS` 重采样
    - 追加一层轻量 `UnsharpMask`
    - 最终保存为 PNG
- `src/backend/routers/generate.py`
  - 只在 `meets_delivery_floor()` 通过后执行 `prepare_delivery_image()`
  - 即：
    - 被 VLM 拦截的图不做升档
    - 通过质检准备交付的图统一变成 4K 成片
- `src/frontend/src/pages/Review.tsx`
  - 下载按钮文案更新为：`下载当前 4K 大图`
