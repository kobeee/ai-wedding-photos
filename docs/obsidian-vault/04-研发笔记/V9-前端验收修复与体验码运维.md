# V9 前端验收修复与体验码运维

> 执行日期：2026-04-04
> 触发：V9 重构部署后用户手动验证发现页面交互异常
> 方法：Codex (GPT-5.4) 全量代码审查 → Claude Code (Opus 4.6) 修复 → VPS 部署
> 结果：8 个 Bug 全部修复，体验码运维脚本上线

---

## 一、VPS 数据清理

验收前先清空旧数据，避免旧数据干扰新流程验证：

```bash
# 容器内清理
docker compose exec -T backend sh -c 'rm -rf /app/data/lumiere.db /app/uploads/* /app/outputs/*'
# 重启 backend，SQLite 自动重建（含 4 个默认 SKU + 10 个默认场景）
docker compose restart backend
```

## 二、Codex 全量审查报告

### 审查范围
- `App.tsx` — 路由定义
- `api.ts` — API 调用层
- `workflow.ts` — 状态管理
- 全部 10 个页面组件

### 发现问题汇总

| # | 严重度 | 文件 | 问题 | 根因 |
|---|--------|------|------|------|
| 1 | **Critical** | Upload.tsx | `proceedToMakeup()` 清空 `selectedSku` | `preserveDownstream=false` 分支把上游选择也清了 |
| 2 | **Critical** | Waiting.tsx | 调用不存在的 `POST /api/orders/{id}/pay` | 后端只有 `/api/pay/mock/create` |
| 3 | **High** | Waiting.tsx | 不读取 URL `:orderId` 参数 | 缺少 `useParams()` |
| 4 | **High** | Waiting.tsx | `fetchLatestOrder` 覆盖新选择 | 总是先 fetch 旧订单 |
| 5 | **High** | PayResult.tsx | 导航到不存在的 `/checkout` | 路由未注册 |
| 6 | **Medium** | SceneSelect.tsx | 正则解析营销文案获取场景数 | 没用 `entitlements.scene_count` |
| 7 | **Medium** | Delivery.tsx | 加载失败显示空白 | `return null` 在 error 之前 |
| 8 | **Medium** | Header.tsx | CTA 跳过 `/plan` 和 `/scenes` | 直接跳 `/upload` |

### 根本原因分析

两个 Critical 问题反映了 **V9 重构时前后端接口对齐不完整**：
- `Upload.tsx` 的状态清理逻辑是从 V8 继承的，V9 新增了 `selectedSku`/`selectedSceneIds` 但清理逻辑没相应调整
- `Waiting.tsx` 的支付调用是按 PRD 设计写的 RESTful 风格（`/orders/{id}/pay`），但后端实际实现是独立的 mock payment 路由

三个 High 问题是 **状态来源优先级设计缺陷**：
- URL 参数 vs localStorage 的优先级没处理好
- 新订单 vs 旧订单的判断逻辑不完善

## 三、修复详情

### Critical #1: selectedSku 被清空

**改动文件**: `Upload.tsx`

`proceedToMakeup()` 和 `handleRemove()` 中的状态清理移除了 `selectedPackage: undefined` 和 `selectedSku: undefined`，保留上游选择（plan/scene/email），只清下游交付状态。

### Critical #2: 支付端点不匹配

**改动文件**: `Waiting.tsx`

```diff
- `/api/orders/${newOrder.order_id}/pay`
- body: JSON.stringify({ provider: 'mock' })
+ '/api/pay/mock/create'
+ body: JSON.stringify({ order_id: newOrder.order_id })
```

### High #3: Waiting 读取 URL orderId

**改动文件**: `Waiting.tsx`

加入 `useParams()` 读取 URL 参数，优先级：URL > workflow > fetchLatest。

### High #4: fetchLatestOrder 逻辑修正

**改动文件**: `Waiting.tsx`

当 workflow 中有 `selectedSku.id + selectedSceneIds` 时，跳过 `fetchLatestOrder()`，直接走创建新订单分支。

### High #5: PayResult /checkout → /plan

**改动文件**: `PayResult.tsx`

全文替换 `navigate('/checkout')` → `navigate('/plan')`。

### Medium #6: 场景数量来源

**改动文件**: `workflow.ts`, `PlanSelect.tsx`, `SceneSelect.tsx`

- `SkuSelection` 新增 `sceneCount?: number` 字段
- `PlanSelect` 保存时写入 `selectedPlan.entitlements.scene_count`
- `SceneSelect` 直接读 `workflow.selectedSku?.sceneCount`，不再正则解析 description

### Medium #7: Delivery 错误状态展示

**改动文件**: `Delivery.tsx`

在 `!previewImage` guard 之前加 `error && !previewImage` 分支，渲染带 header 的错误页面。

### Medium #8: Header CTA 路由

**改动文件**: `Header.tsx`

```diff
- <Link to="/upload" className="header__cta">开始创作</Link>
+ <Link to="/plan" className="header__cta">开始创作</Link>
```

## 四、体验码运维脚本

### 背景

每次需要体验码都要手动 SQL 太麻烦，做了个 VPS 专属脚本。

### 脚本位置（不进仓库）

```
VPS: /opt/apps/wedding-photos/gen-exp-code.sh
```

### 用法

```bash
# 随机码
bash gen-exp-code.sh

# 指定码
bash gen-exp-code.sh LUMIERE2026

# 指定码 + 渠道
bash gen-exp-code.sh MYCODE wechat
```

### 实现原理

通过 `docker compose exec` 在 backend 容器内用 Python sqlite3 直接插入 `experience_codes` 表。
