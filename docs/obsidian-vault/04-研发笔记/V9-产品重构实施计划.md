# V9 产品重构实施计划

> 目标：从"试玩式 AI 生图工具"升级为"订单驱动、邮箱交付的 AI 婚纱摄影棚"
> 基线：V8 管线已终审 A 级（24 case 全通过），AI 生图能力已验证
> 原则：视觉稿先行 → 前后端同步实现 → 部署验证

---

## 一、当前状态 vs 目标态

### 现在的用户旅程

```
Landing → Upload → Makeup → PackageSelect → Waiting → Review
```

- 先上传再选套餐，无效上传成本高
- package/sku 二层用户看不懂
- 没有邮箱绑定，订单找不回来
- Review 只是任务结果页，不是正式交付
- 免费体验是旁路试玩

### 目标用户旅程

```
Landing → PlanSelect(含邮箱) → SceneSelect → Upload → Makeup → Waiting → Delivery
                                                                          ↓
                                                                        Orders(我的订单)
```

- 套餐前置，先成交再定制
- 邮箱前置，用于交付和找回（非注册登录）
- 用户自选景别
- 免费体验 = 正式套餐 + 体验码
- Delivery 是订单交付页（含压缩包）
- Orders 是订单列表/找回入口

### 页面变更矩阵

| 页面 | 动作 | 变化幅度 | 说明 |
|------|------|---------|------|
| Landing | **大改** | ★★★★ | CTA 从"开始上传"改为"选择套餐"，套餐卡片前置，去掉试玩引导 |
| PlanSelect | **新建** | ★★★★★ | 套餐选择 + 邮箱输入 + 体验码入口，取代旧 PackageSelect + Checkout |
| SceneSelect | **新建** | ★★★★ | 根据套餐景数勾选景别，景别分类展示 |
| Upload | **小改** | ★★ | 流程位置后移，StepHeader 更新，逻辑不变 |
| Makeup | **小改** | ★★ | 流程位置后移，StepHeader 更新，逻辑不变 |
| Waiting | **中改** | ★★★ | 从单 task 进度改为订单级多阶段状态展示 |
| Delivery | **大改/新建** | ★★★★★ | 取代 Review，订单交付页：成片浏览 + 压缩包下载 + 重拍入口 |
| Orders | **新建** | ★★★ | 我的订单列表 + 邮箱找回入口 |
| PackageSelect | **废弃** | — | 被 PlanSelect 取代 |
| Checkout | **废弃** | — | 合并进 PlanSelect |
| PayResult | **保留** | ★ | 支付回调页，微调文案 |

## 二、分阶段实施计划

### Phase 1：视觉稿（page.pen）

**目标**：在 page.pen 中完成所有新页面设计，旧页面改版设计

**任务清单**：

- [ ] **T1.1** Landing 改版 — CTA 改为"选择套餐"，套餐卡片组前置到首屏下方，去掉"先上传试试"
- [ ] **T1.2** PlanSelect 新页面 — 四档套餐卡片（免费/399/699/999）+ 邮箱输入框 + 体验码折叠入口 + 确认按钮
- [ ] **T1.3** SceneSelect 新页面 — 景别分类网格（西式/中式/旅拍/影棚/夜景/幻境）+ 已选计数 + 套餐景数约束提示
- [ ] **T1.4** Upload 微调 — StepHeader 步骤名更新（Step 3→Upload），其余保持
- [ ] **T1.5** Makeup 微调 — StepHeader 更新
- [ ] **T1.6** Waiting 改版 — 订单级阶段展示（订单锁定 → 身份提取 → 画面生成 → 身份融合 → 质检 → 打包）
- [ ] **T1.7** Delivery 新页面 — 左：大图浏览，右：缩略图列表 + 订单信息卡片（套餐/景别/张数/邮箱）+ 压缩包下载按钮 + 重拍按钮
- [ ] **T1.8** Orders 新页面 — 订单列表卡片 + 邮箱找回入口 + 订单状态标签
- [ ] **T1.9** StepHeader 组件更新 — 新步骤流：选套餐 → 选景 → 上传 → 妆造 → 等待 → 交付

**设计约束**（来自 [[设计规范]] + `.impeccable.md`）：
- 深色奢华主题：#0A0A0A 底 + #C9A96E 金色
- 每页只做一件事，留白即设计
- 零决策压力，视觉引导替代文字
- 套餐卡片用金色边框高亮主推款（699）

### Phase 2：后端数据层改造

**目标**：Plan 模型落地，订单流程打通，邮箱绑定

**前置依赖**：无（可与 Phase 1 并行）

**任务清单**：

- [ ] **T2.1** 数据库迁移 — 现有 9 张表的改造：
  - `skus` → 重命名为 `plans`，字段对齐 [[婚纱摄影棚重构总方案]] 的 Plan 模型
  - 新增 `scene_catalog` 表（scene_id, category, name, preview_url, description, active, sort_order）
  - 新增 `order_scene_selections` 表（selection_id, order_id, scene_id, sort_order）
  - 新增 `experience_codes` 表（code_id, code, channel, status, bound_email, bound_order_id, 时间戳）
  - `orders` 表加 `email` 字段 + `experience_code_id` 字段 + `delivery_status` 字段
  - 废弃旧 `tasks` 表（已被 generation_batches + deliverables 取代）
  - `identities` 表加 `email` 字段（kind=email 时填充）

- [ ] **T2.2** Scene Catalog 种子数据 — 初始化景别目录（对应 10 套 brief 的场景）：
  | category | scenes |
  |----------|--------|
  | 西式 | iceland_coast, french_street |
  | 中式 | chinese_courtyard, chinese_night |
  | 旅拍 | travel_steps, travel_arch |
  | 影棚 | minimal_studio |
  | 夜景 | cyber_neon, star_fairylights |
  | 幻境 | fantasy_canopy |

- [ ] **T2.3** Plan 种子数据 — 四档套餐：
  ```
  trial_free:   price=0,   scene_quota=1, promised_photos=3,  rerun_quota=0, requires_code=true
  starter_399:  price=399,  scene_quota=3, promised_photos=24, rerun_quota=1
  memory_699:   price=699,  scene_quota=5, promised_photos=40, rerun_quota=2
  archive_999:  price=999,  scene_quota=7, promised_photos=56, rerun_quota=3
  ```

- [ ] **T2.4** 新 API 端点：
  - `GET /api/plans` — 套餐列表
  - `GET /api/scenes` — 景别目录
  - `POST /api/orders` — 创建订单（plan_id + email + scene_ids + experience_code?）
  - `GET /api/orders/{order_id}` — 订单详情
  - `GET /api/orders/lookup?email=xxx` — 邮箱找回订单
  - `POST /api/orders/{order_id}/start` — 启动生产（替代旧 POST /api/generate）
  - `GET /api/orders/{order_id}/deliverables` — 交付物列表
  - `POST /api/experience-codes/verify` — 体验码校验

- [ ] **T2.5** Session 改造 — Bearer token → HttpOnly Cookie（参见 [[收费闭环系统改造实施方案]] Phase 0）

- [ ] **T2.6** 邮箱绑定逻辑 — 创建订单时绑定邮箱到 identity，同一邮箱关联同一 identity

### Phase 3：前端实现

**目标**：基于 Phase 1 视觉稿 + Phase 2 后端 API，实现新用户旅程

**前置依赖**：Phase 1（视觉稿定稿）+ Phase 2（API 可用）

**任务清单**：

- [ ] **T3.1** 路由重构 — App.tsx 更新路由表：
  ```
  /                → Landing（改版）
  /plan            → PlanSelect（新）
  /scenes          → SceneSelect（新）
  /upload          → Upload（微调）
  /makeup          → Makeup（微调）
  /waiting/:orderId → Waiting（改版）
  /delivery/:orderId → Delivery（新，取代 /review）
  /orders          → Orders（新）
  /pay/result      → PayResult（保留）
  ```

- [ ] **T3.2** Landing 页改版 — 按视觉稿实现，CTA 指向 /plan

- [ ] **T3.3** PlanSelect 页实现 — 套餐卡片 + 邮箱输入 + 体验码 + 调用 POST /api/orders

- [ ] **T3.4** SceneSelect 页实现 — 景别网格 + 勾选约束 + 写入订单

- [ ] **T3.5** Upload/Makeup 微调 — StepHeader 更新，order_id 上下文透传

- [ ] **T3.6** Waiting 页改版 — 基于订单状态轮询，展示多阶段进度

- [ ] **T3.7** Delivery 页实现 — 成片浏览 + 订单信息 + 下载 + 重拍

- [ ] **T3.8** Orders 页实现 — 订单列表 + 邮箱找回

- [ ] **T3.9** StepHeader 组件重构 — 新 6 步流程

- [ ] **T3.10** 废弃旧页面 — 删除 PackageSelect.tsx + Checkout.tsx，清理旧路由

### Phase 4：管线对接与生产流程改造

**目标**：订单驱动生产，多景多张批量生图

**前置依赖**：Phase 2（订单层）+ V8 管线（已就绪）

**任务清单**：

- [ ] **T4.1** 订单编排层 — 根据订单的 scene_selections + plan 权益，生成 shot_plan：
  - 每个景分配 N 张（couple/bride/groom 按比例）
  - 写入 generation_batches

- [ ] **T4.2** 批量生产适配 — 现有 generate.py 的单次生成逻辑改为循环 shot_plan：
  - 按 scene + gender + composition 遍历
  - 每张图独立走 V8 三轮管线（R1→R2→R3）
  - 结果写入 deliverables

- [ ] **T4.3** 生产状态推送 — 批次进度实时更新到 generation_batches，前端轮询展示

- [ ] **T4.4** 交付打包 — 订单所有 deliverables 通过质检后：
  - 生成压缩包
  - 写入 deliverable_packages
  - 更新订单 delivery_status

### Phase 5：部署与验证

**目标**：全链路上线，用户亲测

**任务清单**：

- [ ] **T5.1** 数据库迁移脚本 — SQLite 迁移，保留旧数据兼容
- [ ] **T5.2** Docker 镜像重建 — 前后端重新 build
- [ ] **T5.3** VPS 部署 — rsync + docker compose up
- [ ] **T5.4** 冒烟测试 — 完整走一遍新旅程：选套餐 → 填邮箱 → 选景 → 上传 → 妆造 → 等待 → 交付
- [ ] **T5.5** 体验码测试 — 免费套餐 + 体验码校验链路

## 三、暂缓项（Phase 2+）

以下功能不在本轮范围，后续独立迭代：

| 功能 | 原因 |
|------|------|
| 支付宝接入 | 先验证免费 + 体验码链路，支付单独做 |
| 8K 超分 | 当前 4K 交付足够，超分是增值功能 |
| 邮件自动投递 | 先做压缩包生成 + 页面下载，邮件发送后补 |
| 魔法笔刷/局部重绘 | Phase 2 功能 |
| AI 强拆分 8 阶段管线 | 当前 V8 三轮管线已验证 A 级，强拆分是性能优化方向，不阻塞上线 |
| 客服后台 | 非 MVP |

---

## 四、技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 多景批量生图耗时长 | 40 张图 ~50 分钟 | Waiting 页做好预期管理，分阶段展示进度 |
| 邮箱找回跨设备 | 无密码，靠邮箱匹配 | 邮箱 + 订单号双因子查询 |
| SQLite 并发瓶颈 | 多用户同时生产 | 当前用户量可承受，后续可迁 PostgreSQL |
| 旧数据兼容 | 已有 tasks 表数据 | 迁移脚本保留旧表只读，新流程走新表 |

## 五、执行节奏建议

```
Session N+1:  Phase 1（视觉稿）+ Phase 2（后端数据层）并行
Session N+2:  Phase 3（前端实现）
Session N+3:  Phase 4（管线对接）+ Phase 5（部署验证）
```

Phase 1 和 Phase 2 无依赖可并行。Phase 3 等视觉稿和 API 都就绪后开始。Phase 4 和 Phase 5 收尾。
