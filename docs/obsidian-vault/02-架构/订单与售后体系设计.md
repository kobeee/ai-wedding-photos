# 订单与售后体系设计

## 一句话结论

这个项目后续不能只做“支付成功就开始生成”。

更稳的主线应该是：

**匿名身份 -> 订单 -> 支付 -> 权益生效 -> 生成批次 -> 交付清单 -> 售后工单**

也就是说，真正的业务主键不该只是 `user_id` 或 `task_id`，而应该是 **order**。

---

## 为什么要以订单为中心

当前站点的真实业务不是“用户发起一个生图任务”，而是“用户购买一份婚纱摄影交付承诺”。

按现有产品文档，系统对用户承诺的不是单次调用，而是：

- 套餐张数
- 分辨率规格
- 重拍次数
- 下载/保存时效
- 不足补偿
- 大面积故障时的延期或退款

这些都属于订���语义，不属于单个生成任务语义。

所以：

- `identity/session` 解决“这是谁”
- `order` 解决“买了什么、承诺什么”
- `generation job` 解决“这一轮生成发生了什么”
- `after-sale ticket` 解决“交付后怎么处理争议和补救”

## 核心设计原则

## 1. 订单是唯一承诺载体

不要把套餐规则散落在：

- 前端文案
- prompt
- task 内存变量

订单创建时就要把权益快照固化下来。

这样后面即使：

- 改价
- 改套餐文案
- 调整默认重拍规则

也不会影响历史订单履约。

## 2. 支付和生成解耦

支付成功只代表“权益已生效”，不代表“任务必须立刻启动”。

回调只做：

- 验签
- 幂等更新支付状态
- 激活订单权益

开始生成应该是显式动作：

- 前端拉起
- 或后端安全异步调度

## 3. 售后不是例外，而是正式状态

婚纱摄影这种高感知产品，售后不是边角逻辑。

“重拍、微调、补张、延期、退款”应该是状态机的一部分，而不是客服手动查日志。

## 4. 交付单位是订单，不是单张图片

用户感知的是：

- 这一单够不够用
- 有没有达到承诺张数
- 不满意怎么办

所以售后判断的第一层是订单履约，不是单张技术分。

## 推荐状态机

## 一、订单状态

建议拆成 3 条并行状态，而不是塞进一个 `status` 字段。

### 1. payment_status

- `unpaid`
- `pending`
- `paid`
- `partially_refunded`
- `refunded`
- `failed`
- `expired`

### 2. fulfillment_status

- `not_started`
- `queued`
- `generating`
- `partially_delivered`
- `delivered`
- `blocked`
- `cancelled`

### 3. service_status

- `normal`
- `in_aftercare`
- `awaiting_user`
- `resolved`
- `closed`

如果为了第一期简化，也至少保留：

- `order_status`: `created / pending_payment / paid / generating / delivered / aftersale / closed / refunded`

但内部逻辑最好仍按三层理解。

---

## 二、生成批次状态

- `queued`
- `running`
- `quality_check`
- `completed`
- `partially_completed`
- `failed`
- `cancelled`

批次与订单的关系：

- 一笔订单至少有 1 个 `initial` 批次
- 用户使用重拍额度时新增 `rerun` 批次
- 客服介入精修时新增 `manual_retouch` 批次

---

## 三、售后工单状态

- `open`
- `triaging`
- `waiting_user`
- `processing`
- `resolved`
- `rejected`
- `closed`

## 用户视角的完整主线

## 阶段 1：匿名进入

1. 用户进入首页，不登录
2. 上传照片时创建 `identity + session`
3. 用户完成试妆和视觉套餐选择

## 阶段 2：创建订单

1. 用户确认购买 SKU
2. 服务端创建 `order`
3. 订单快照固化：
   - 价格
   - promised_photos
   - 重拍额度
   - 保存周期
   - 下载规格
   - 补偿规则版本

## 阶段 3：支付

1. 前端拉起支付宝/微信
2. 平台异步回调后端
3. 后端验签并更新 `payment_transaction`
4. 订单变为 `paid`

## 阶段 4：履约

1. 创建 `generation_batch(initial)`
2. 生成多张候选图
3. 每张图写入结构化元数据
4. 汇总出本单交付清单
5. 订单进入：
   - `delivered`
   - 或 `partially_delivered`
   - 或 `blocked`

## 阶段 5：交付与售后

1. 用户在审片页查看本单交付结果
2. 可以下载、选片、申请重拍、申请微调
3. 系统根据剩余额度与原因创建售后工单
4. 售后处理结果回写订单

## 订单怎么跟踪

要跟踪订单，核心不是只记“支付成功”，而是记下面 5 层证据链。

## 1. 商业证据

- 买的是哪个 SKU
- 实付多少
- 当时承诺什么权益

## 2. 支付证据

- 哪个渠道
- 哪个交易号
- 哪次通知确认成功

## 3. 生产证据

- 一共发起了几轮生成
- 每轮生成了多少图
- 每张图的得分、修复、拦截原因

## 4. 交付证据

- 最终交付了几张
- 哪些规格可下载
- 哪些图被用户选中或下载

## 5. 售后证据

- 用户提出什么问题
- 是否消耗重拍/微调额度
- 最终怎么处理

缺少其中任何一层，后面都会出现“付了钱但说不清发生了什么”。

---

## 售后应该怎么设计

售后不要只理解成退款。

对这个项目，售后至少分 4 类：

## 1. 技术售后

系统侧问题：

- 人脸明显不像
- 手部穿模
- 双人比例崩
- 结果数量不足
- 下载资源损坏

处理原则：

- 系统责任不消耗用户重拍额度
- 优先补生成或补交付
- 必要时进入退款

## 2. 审美售后

偏主观问题：

- 妆容不喜欢
- 场景氛围不喜欢
- 服装风格想调整

处理原则：

- 走套餐内重拍或微调额度
- 超额后可加购

## 3. 履约售后

订单承诺未达成：

- 72 小时未交付
- promised_photos 不足
- 付费规格未解锁

处理原则：

- 订单标记 `in_aftercare`
- 自动触发补偿或人工介入

## 4. 退款售后

最终无法履约或用户依法主张退款。

处理原则：

- 支持部分退款和全额退款
- 与实际交付数量、已下载高规格资源、协议条款联动

---

## 售后判断规则建议

建议先把售后规则写成后端策略，而不是客服口头约定。

## 系统责任

以下情况不扣用户额度：

- 支付成功但订单未启动
- 第三方模型不可用导致延期
- 生成图全部未达最低可交付阈值
- 已承诺 10 张但最终不足最低交付线

## 用户主观修改

以下情况消耗额度：

- 更换场景风格
- 更换妆造方向
- 要求重新生成一轮不同氛围

## 灰区情况

以下情况进入人工或半自动审核：

- “像本人但不够高级”
- “氛围很好但想更瘦一点”
- “部分图满意，部分图不满意”

建议为工单增加 `reason_code`，不要直接写自然语言：

- `identity_mismatch`
- `physical_artifact`
- `style_preference`
- `quantity_shortfall`
- `sla_breach`
- `download_problem`
- `refund_request`

## 第一版最该落地的数据库表

你现在已有 `users` 和 `tasks`，第一版建议最少补下面 6 张。

1. `identities`
2. `sessions`
3. `orders`
4. `payment_transactions`
5. `generation_batches`
6. `aftersale_tickets`

如果想进一步把交付追踪做实，再补：

7. `deliverables`
8. `assets`
9. `order_events`

其中 `order_events` 很关键，用来做审计流：

- `order_created`
- `payment_created`
- `payment_succeeded`
- `generation_started`
- `photo_rejected`
- `delivery_completed`
- `ticket_opened`
- `refund_completed`

## 前端流程应该怎么改

当前主线还是：

上传 -> 试妆 -> 选套餐 -> 直接生成

建议改成：

上传 -> 试妆 -> 选视觉套餐 -> 选 SKU -> 创建订单 -> 支付 -> 订单确认页 -> 开始生成 -> 审片交付 -> 售后入口

注意两点：

## 1. `package_id` 和 `sku_id` 分开

- `package_id` = 视觉风格
- `sku_id` = 商品与权益

不能再混成一个“套餐”。

## 2. Review 页要从“任务结果页”升级成“订单交付页”

Review 页未来应该展示：

- 当前订单号
- 已交付张数 / 承诺张数
- 下载规格
- 剩余重拍次数
- 售后入口
- 数据保留截止时间

---

## 最小落地顺序

建议按这个顺序，不要一口气做全量售后系统。

## Phase 1

- 匿名账户改为服务端 cookie session
- 新增 `orders`、`payment_transactions`
- `POST /api/generate` 改为必须校验 `order_id`
- 支付成功后才能开始生成

## Phase 2

- 任务状态与单张交付元数据持久化
- Review 页升级为订单交付页
- 新增重拍额度与订单履约统计

## Phase 3

- 新增 `aftersale_tickets`
- 支持重拍申请、下载异常、退款申请
- 支持系统责任自动判定是否扣额度

## Phase 4

- 可选绑定邮箱/手机号
- 跨设备找回订单
- 客服后台检索订单与工单

---

## 最终判断

对这个项目，真正应该设计的不是“支付模块”，而是：

**一个可签约、可追踪、可补救的婚纱摄影订单系统。**

只要你未来要收钱、承诺交付、支持重拍、处理退款和售后，就必须让：

- 身份
- 订单
- 支付
- 交付
- 售后

形成同一条结构化主线。

这条主线里，**订单是中心，支付是触发器，生成是履约过程，售后是履约延伸。**
