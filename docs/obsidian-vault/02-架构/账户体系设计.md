# 账户体系设计

## 结论

本项目不适合做传统注册登录，但也不应该继续停留在“前端 localStorage 持久化 `user_id + session_token`”这一层。

更合适的方向是：

**默认匿名账户（guest account） + 服务端会话（session） + 可选找回/绑定（magic link）**

即：

1. 用户首次访问不登录
2. 首次上传时由服务端创建匿名用户与会话
3. 会话默认保存在 `HttpOnly + Secure + SameSite=Lax` Cookie，而不是前端 `localStorage`
4. 全部用户资产（上传图、结果图、任务状态、订单）都挂在匿名账户下面
5. 仅在用户需要“跨设备查看 / 延长保存 / 支付后交付 / 售后找回”时，提示绑定手机号或邮箱

这套方案比“强制注册登录”转化更高，也比“完全无账户”更能承接高价值生成结果与隐私责任。

---

## 为什么不建议完全无账户

婚纱摄影���是低价值的一次性玩具产品，用户会产生以下真实诉求：

- 想隔天回来继续看结果
- 想在手机和电脑之间切换
- 想支付后继续补生成或复看
- 想联系客服时能快速定位自己的作品
- 想确认隐私数据在谁的名下、何时销毁

如果完全无账户，只靠前端页面内状态：

- 清缓存、换浏览器、换设备后，作品直接丢失
- 用户无法自助找回
- 客服无法稳定定位资产归属
- 订单、支付、交付、售后无法闭环

所以“不想让用户登录”是对的，但“不做身份层”不对。

## 推荐方案

## Phase 1：匿名账户为主

### 用户体验

- 首页浏览：公开，无需身份
- 上传照片：自动创建匿名账户
- 试妆、生成、审片：自动沿用当前匿名会话
- 用户感知：全程不出现“注册 / 登录”字样

### 技术设计

- `guest_user`
  - 内部主键 `user_id` 使用服务端生成 UUID
  - 不允许客户端显式传入并复用
- `session`
  - 使用高熵随机 opaque token
  - 仅存服务端哈希值或直接存 DB 明文后续再升级哈希
  - 通过 `Set-Cookie` 写入浏览器
- `workflow`
  - 前端只保存非敏感流程状态，如当前步骤、选中的套餐、试妆偏好
  - 不保存真正的鉴权凭证

### 适用范围

- 未支付前的完整体验
- 24 小时内的照片与结果访问
- 同设备连续使用

---

## Phase 2：可选找回 / 轻绑定

触发时机：

- 查看结果后点击“保存我的作品”
- 支付前后
- 用户准备离开页面
- 客服找回场景

推荐方式：

- 邮箱 magic link
- 手机验证码

不要做：

- 用户名密码注册
- 社交登录首屏强打断

绑定后得到的能力：

- 跨设备找回
- 延长保留期
- 支付订单关联
- 售后记录归档

这一步是“升级账户”，不是“强制登录”。

---

## 账户对象模型建议

建议拆成 4 层，而不是把 `user_id` 和 `session_token` 当成全部账户体系：

### 1. identity

- `identity_id`
- `kind`: `guest` / `email` / `phone`
- `email` 或 `phone`（可空）
- `bound_at`
- `status`

### 2. session

- `session_id`
- `identity_id`
- `token_hash`
- `device_info`
- `ip`
- `created_at`
- `expires_at`
- `last_seen_at`

### 3. asset

- `asset_id`
- `identity_id`
- `type`: `upload` / `makeup_preview` / `final_output`
- `storage_key`
- `retention_expires_at`
- `created_at`

### 4. job / order

- `job_id`
- `identity_id`
- `package_id`
- `status`
- `payment_status`
- `created_at`

这样后面接支付、延长存储、客服检索时不会反复推翻数据模型。

## 当前实现存在的问题

以下问题决定了当前实现不能直接作为长期账户体系。

### 1. `user_id` 可被客户端带回上传接口复用

现状：

- `src/frontend/src/pages/Upload.tsx` 会把已有 `workflow.userId` 再次提交给 `/api/upload`
- `src/backend/routers/upload.py` 允许客户端传入 `user_id`
- `src/backend/models/database.py` 的 `create_user()` 使用 `INSERT OR IGNORE`
- 如果该 `user_id` 已存在，会直接读回该用户已有 `session_token`

风险：

- 任何知道 `user_id` 的人，都有机会通过公开上传接口换回该账户 token
- 这相当于“user_id 既是账户编号，也是找回凭证入口”

结论：

- `user_id` 不能再由客户端指定
- “续传”必须依赖现有受保护 session，而不是明文传 `user_id`

### 2. 上传图与结果图现在是静态公开路径

现状：

- `middleware/auth.py` 把 `/api/files/` 视为公开路径
- `main.py` 通过 `StaticFiles` 直接挂出 uploads/outputs 目录

风险：

- 只要知道 URL 就能直接访问用户照片
- 与“所有 API 必须鉴权”“用户面部数据 24 小时内自动销毁”的隐私叙事不一致

建议：

- 不再直挂静态目录给公网
- 改为受保护下载接口：
  - 校验当前 session 是否拥有该资源
  - 再流式返回文件
- 或者短期使用带过期时间的签名 URL

### 3. 鉴权凭证保存在 `localStorage`

问题：

- 容易被前端脚本读取
- 与未来的风控、续期、设备管理难兼容

建议：

- 改为 `HttpOnly Cookie`
- 前端不再手工拼 `Authorization` header
- 同站请求自动带 cookie

### 4. 任务状态主要在内存中

现状：

- 数据库里有 `tasks` 表
- 但运行链路仍主要依赖进程内 `_tasks` / `_task_owners`

影响：

- 服务重启后任务状态不可恢复
- “账户下历史任务”无法稳定查询

建议：

- 任务创建、轮询、结果查询全面切回 DB
- 会话只负责“你是谁”，任务表负责“你做过什么”

## 最适合当前项目的落地策略

## 方案名称

**无登录首屏 + 匿名会话闭环 + 结果页轻绑定**

## 用户路径

1. 用户进入首页，不登录
2. 上传照片时，服务端创建匿名身份与 session cookie
3. 后续试妆/生成/审片全部自动识别当前用户
4. 结果页提示：
   - “保存 7 天”
   - “发到邮箱”
   - “绑定手机号后可跨设备继续查看”
5. 支付后要求最轻绑定方式之一

## 这样做的好处

- 首屏转化高
- 全流程基本无登录感
- 能承接支付和交付
- 能满足隐私审计
- 能给客服和售后一个稳定索引

---

## 不建议的方案

### A. 完全无账户、只靠本地状态

不建议原因：

- 无法找回
- 无法跨设备
- 无法稳定承接支付后交付
- 不利于隐私与访问控制

### B. 一上来强制手机号登录

不建议原因：

- 上传前就打断
- 对尝鲜用户转化伤害大
- 与产品“仪式感、低决策压力、无技术感”冲突

### C. 直接上 JWT + 完整注册系统

不建议原因：

- 当前阶段明显过重
- 你现在主要问题不是“身份类型太弱”，而是“会话边界太松”

---

## 近期改造优先级

### P0

1. 禁止客户端传 `user_id` 创建或续用账户
2. 上传/结果文件从公开静态路径改为受保护访问
3. 会话从 `localStorage Bearer token` 迁移到 `HttpOnly Cookie`

### P1

1. DB 持久化任务状态，替代进程内 `_tasks`
2. 增加匿名会话续期与过期策略
3. 为结果页增加“发到邮箱找回 / 绑定手机号”能力

### P2

1. 建立 identity / session / asset / order 四层模型
2. 支付成功后自动绑定订单与身份
3. 加入客服检索、删除申请、延长保留期能力

---

## 最终判断

对于这个项目，“不让用户登录”应该被翻译成：

**不让用户感受到登录阻力**

而不是：

**没有身份、没有会话、没有资源归属**

最优解不是“取消账户体系”，而是“把账户体系做成无感的匿名账户体系”，并在真正需要的时候再升级为轻绑定账户。
