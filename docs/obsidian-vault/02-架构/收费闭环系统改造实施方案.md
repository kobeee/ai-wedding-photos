# 收费闭环系统改造实施方案

## 一句话结论

当前项目若要进入真实收费，主线必须从：

`上传 -> 试妆 -> 选套餐 -> 直接生成`

改为：

`匿名身份 -> 视觉主题 -> 销售 SKU -> 创建订单 -> 支付 -> 权益生效 -> 生成批次 -> 订单交付页 -> 售后`

这次改造的核心不是“接支付按钮”，而是把：

- 身份
- 订单
- 支付
- 履约
- 交付

统一到同一条业务主线里。

---

## 本轮已拍板决策

### 商品结构

- 视觉主题与销售商品拆开：
  - `package_id` 表示视觉风格 / 场景主题
  - `sku_id` 表示销售商品 / 权益包
- 套餐先按以下结构试跑：
  - `399`：`3 景 / 24 张`
  - `699`：`5 景 / 40 张`
  - `999`：`7 景 / 56 张`
- `699` 作为唯一主推款
- 免费体验保留，先出 `3 张`样片

### 交付口径

- 当前对外统一口径为 `4K` 交付，不再承诺 `8K`
- 免费体验不把“水印”作为核心防护手段
- 免费体验与风控主要依赖：
  - 张数限制
  - 规格限制
  - 结果保留期
  - 订单与身份绑定
  - 不开放完整权益

### 技术原则

- 技术选型以“合理、高效、可扩展、可维护”为准
- 第一阶段优先形成最小收费闭环，不为了“架构完美”拖慢上线
- 支付第一期优先支付宝
- 订单是唯一履约主键

---

## 目标与非目标

## 目标

- 建立真实收费所需的订单与支付底座
- 让权益、张数、重拍额度、保留期从文案变成后端真源
- 让 Review 页从“任务结果页”升级为“订单交付页”
- 为后续售后、增购、轻绑定、退款预留稳定对象模型

## 非目标

- 本阶段不追求一次把微信支付、会员订阅、实体冲印全部做完
- 本阶段不追求完整客服后台
- 本阶段不在 prompt 层写任何定价和权益规则

---

## 现状问题

## 1. 身份层还停留在试玩态

当前实现仍以：

- `user_id + session_token`
- 前端 `localStorage`
- `Authorization: Bearer`

为主。

这套方案对试玩够用，但对收费闭环不够稳。

## 2. 订单层缺失

当前数据库只有：

- `users`
- `tasks`

没有：

- `orders`
- `payment_transactions`
- `generation_batches`
- `deliverables`

所以无法稳定回答：

- 用户买了什么
- 订单承诺多少张
- 已交付多少张
- 重拍额度还剩多少
- 是否该退款或补偿

## 3. 生成链路没有支付闸门

当前前端会在 Waiting 页直接调用 `POST /api/generate`。

这意味着：

- 还没有“先下单、再支付、再履约”的强约束
- 无法形成正式收费流程

## 4. 交付承诺与工程能力未对齐

当前工程默认每单仅产 `4` 张成片位。

若要对外卖：

- `24 / 40 / 56`

则必须把履约能力从“单 task 产几张”升级为“订单内多批次累计交付”。

## 目标架构

## 核心对象

### 1. `identity`

谁拥有这笔资产。

建议字段：

- `identity_id`
- `kind`: `guest` / `email` / `phone`
- `email`
- `phone`
- `status`
- `created_at`

### 2. `session`

哪个设备当前持有会话。

建议字段：

- `session_id`
- `identity_id`
- `token_hash`
- `device_info`
- `ip`
- `created_at`
- `expires_at`
- `last_seen_at`

### 3. `sku`

卖什么。

第一批建议：

- `trial_free`
- `starter_399`
- `memory_699`
- `archive_999`

建议字段：

- `sku_id`
- `name`
- `price`
- `currency`
- `active`
- `entitlements_json`

`entitlements_json` 建议至少包含：

- `promised_photos`
- `scene_count`
- `photo_mix`
- `rerun_quota`
- `repaint_quota`
- `retention_days`
- `delivery_specs`
- `preview_policy`

### 4. `order`

唯一承诺载体。

建议字段：

- `order_id`
- `identity_id`
- `sku_id`
- `package_id`
- `amount`
- `currency`
- `payment_status`
- `fulfillment_status`
- `service_status`
- `entitlement_snapshot_json`
- `created_at`
- `paid_at`
- `expired_at`
- `closed_at`

### 5. `payment_transaction`

支付证据链。

建议字段：

- `payment_id`
- `order_id`
- `provider`
- `provider_trade_no`
- `provider_buyer_id`
- `amount`
- `status`
- `notify_payload`
- `paid_at`
- `created_at`

### 6. `generation_batch`

订单内的一轮实际生产。

建议字段：

- `batch_id`
- `order_id`
- `batch_type`: `preview` / `initial` / `rerun` / `manual_retouch`
- `initiated_by`: `system` / `user` / `support`
- `status`
- `scene_index`
- `started_at`
- `completed_at`
- `failure_reason`

### 7. `deliverable`

最终交付对象。

建议字段：

- `deliverable_id`
- `order_id`
- `batch_id`
- `asset_id`
- `scene_index`
- `photo_role`: `couple` / `bride` / `groom` / `detail`
- `photo_status`: `delivered` / `failed` / `needs_human`
- `quality_score`
- `delivery_tier`: `preview` / `4k`
- `created_at`

## 关键产品规则映射

## 1. 免费体验

免费体验先产 `3 张`样片，不默认加水印。

限制方式：

- 只给少量结果
- 不解锁整套张数
- 不解锁正式重拍额度
- 保留期更短
- 默认不支持批量导出整单

## 2. 付费套餐

先按下面结构建 SKU：

### `starter_399`

- `scene_count = 3`
- `promised_photos = 24`
- `photo_mix = { couple: 15, bride: 6, groom: 3 }`

### `memory_699`

- `scene_count = 5`
- `promised_photos = 40`
- `photo_mix = { couple: 25, bride: 10, groom: 5 }`

### `archive_999`

- `scene_count = 7`
- `promised_photos = 56`
- `photo_mix = { couple: 35, bride: 14, groom: 7 }`

### 共性规则

- 单景默认 `8 张`
- “景”定义为场景组，而不是强制完全不同世界观
- 默认交付规格为 `4K`
- 订单级履约目标是总张数，不要求一次批次全产完

## 前后端改造方案

## Phase 0：身份与安全底座

### 后端

- 禁止客户端在 `/api/upload` 显式传入 `user_id` 复用账户
- 上传时由服务端创建匿名身份和 session
- session 从 Bearer token 迁到 `HttpOnly + Secure + SameSite=Lax` Cookie
- 文件访问不再走公开静态目录
- 新增受保护下载接口或短期签名 URL

### 前端

- workflow 仅保存流程状态，不保存真正鉴权凭证
- API 请求从手工拼 `Authorization` 改为浏览器自动带 cookie

### 完成标准

- 用户首屏仍无登录感
- 同设备可连续使用
- 资源不再能被猜 URL 直接访问

## Phase 1：订单层落地

### 新表

- `identities`
- `sessions`
- `skus`
- `orders`
- `generation_batches`
- `deliverables`
- `payment_transactions`

### 新接口

- `GET /api/skus`
- `POST /api/orders`
- `GET /api/orders/{order_id}`
- `POST /api/orders/{order_id}/start`

### 关键规则

- `package_id` 只表示视觉主题
- `sku_id` 只表示销售商品
- 创建订单时把权益快照固化进 `entitlement_snapshot_json`

### 完成标准

- 能根据 `package_id + sku_id` 生成合法订单
- 历史订单不受后续改价和改套餐影响

## Phase 2：支付宝闭环

### 新接口

- `POST /api/pay/alipay/create`
- `POST /api/pay/alipay/notify`

### 支付回调规则

- 只负责验签
- 幂等更新订单支付状态
- 记录 `payment_transaction`
- 不直接同步触发整条 AI 流水线

### 前端流程

改为：

`上传 -> 试妆 -> 选视觉主题 -> 选 SKU -> 创建订单 -> 拉起支付宝 -> 支付结果页 -> 轮询订单 -> 开始生成`

### 完成标准

- 能真实下单
- 能真实支付
- 支付成功后权益生效
- 未支付订单不能生成

## Phase 3：履约对象化

### 后端

- `/api/generate` 改为订单内部履约入口
- 所有生成记录到 `generation_batches`
- 所有结果图记录到 `deliverables`
- 重拍不再是“再点一次生成”，而是新增 `rerun` 批次

### 前端

- Waiting 页基于订单状态和批次状态展示
- Review 页升级为订单交付页，展示：
  - 订单号
  - 套餐权益
  - 已交付张数
  - 批次状态
  - 可用微调 / 重拍额度

### 完成标准

- 订单履约可追踪
- 交付和售后都有稳定对象可挂

## Phase 4：大张数履约能力

### 核心做法

- 不再把整单视为一个一次性 task
- 按场景组或批次逐步产出
- 最终由订单聚合总交付张数

### 目的

- 支撑 `24 / 40 / 56` 套餐
- 为后续“补某一景”“替换某一景”提供技术基础

### 完成标准

- 订单承诺张数与真实交付能力一致
- 收费前满足 [[战略决策稿]] 中“工程与商业一张皮”要求

## 实施顺序

如果按“最快形成可收费闭环”的顺序，建议这样排：

1. 身份与 session 改造
2. 文件访问收口
3. 订单表与 SKU 表
4. 订单 API
5. 支付宝创建单与回调
6. 前端 checkout / pay-result
7. 生成入口切到订单
8. Review 页升级为订单交付页
9. 再做大张数履约能力

## 上线策略

当前更适合：

- 小范围灰度
- 最小真实收费闭环
- 最小真实交付闭环

不适合：

- 一开始就公开大推
- 一开始就并行做支付宝 + 微信 + 会员 + 售后后台

## 最终判断

对当前项目，最重要的不是“先把套餐文案写漂亮”，而是：

**先把订单、支付、履约、交付四层打通，让用户买到的承诺真的可交付。**

只有这件事做实后：

- `399 / 699 / 999`
- `24 / 40 / 56`
- 免费体验
- 重拍和微调额度

这些商业设计才有真正落地的地基。
