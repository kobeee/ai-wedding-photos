# 支付接入方案

> 面向当前 AI 婚纱摄影 Web 站点的第一版支付接入调研与落地建议。

---

## 一句话结论

当前站点最适合的第一期方案不是直接上“微信 JSAPI + 公众号”，而是：

- 支付宝：网页支付能力
- 微信支付：移动端用 H5 支付，PC 端用 Native 扫码支付

这样可以覆盖现有站点的 Web 访问场景，不需要先做公众号、小程序或 App 容器。

---

## 当前站点现状

代码现状说明了两个事实：

1. 前端已经有价格展示，但还没有真正的售卖商品模型
2. 后端会在用户选完视觉套餐后直接创建生成任务，没有支付闸门

现有证据：

- Landing 页已经展示 `¥0 / ¥99 / ¥599` 三档方案，但只是展示文案，还没有订单对象和支付动作
- PackageSelect 页只记录“视觉套餐”选择，然后直接跳到 Waiting
- Waiting 页会直接调用 `POST /api/generate`
- 后端 SQLite 目前只有 `users` 和 `tasks` 两张核心表，没有 `orders` / `payments` / `refunds`

这意味着支付接入不能只加一个按钮，而是要先把：

- 视觉套餐（风格）
- 销售 SKU（免费体验 / 基础套餐 / SVIP）
- 订单状态
- 支付状态
- 权益解锁

这五件事拆开。

---

## 推荐的产品组合

## 支付宝

对于现在这个站点，支付宝适合走网页支付能力。

原因：

- 你当前是网站，不是原生 App
- 支付宝开放平台对“网页/移动应用”提供支付接入能力
- 官方接入流程就是创建应用、配置密钥/网关、审核上线

工程侧建议：

- 移动浏览器走手机网站支付
- PC 浏览器走电脑网站支付

这两个能力组合起来，能覆盖你当前 H5 + 桌面浏览器的访问场景。

## 微信支付

微信支付需要按访问容器选产品：

- 微信外的手机浏览器：H5 支付
- PC 网页：Native 支付（展示二维码，用户微信扫一扫）
- 微信内置浏览器：JSAPI 支付

对当前站点的建议：

- 第一版先做 H5 + Native
- 不优先做 JSAPI

因为 JSAPI 适用于“微信客户端内部浏览器”，并且通常要绑定公众号能力；而你当前站点并不是公众号闭环产品。

---

## 为什么不建议第一期先做 JSAPI

JSAPI 的体验确实最好，但它不是当前这个项目的最低成本路径。

原因：

- 你现在的主产品形态是公开 Web 站
- 目前代码里没有公众号体系，也没有围绕微信内访问做页面逻辑分支
- 你第一期最需要的是“支付成功后自动解锁并允许生成”

如果第一期就上 JSAPI，会把问题扩大成：

- 公众号主体准备
- 微信内 H5 兼容
- openid 获取
- 支付目录与授权域名配置
- 微信内外两套收银流

这对当前项目不是最短路径。

---

## 第一期开工建议

建议按下面顺序接：

1. 支付宝网页支付
2. 微信 Native 支付
3. 微信 H5 支付
4. 视流量情况再补微信 JSAPI

这么排的原因：

- 支付宝网页支付最贴合你当前“网站下单”形态
- 微信 Native 最适合桌面端，用户认知成本低
- 微信 H5 解决非微信内手机浏览器支付
- JSAPI 放第二阶段，等公众号/私域链路明确后再上

---

## 对当前业务流程的改造建议

当前流程是：

上传 → 试妆 → 选视觉套餐 → 直接生成

支付接入后建议改成：

上传 → 试妆 → 选视觉套餐 → 选商品 SKU → 创建订单 → 拉起支付 → 支付成功 → 创建生成任务

关键变化：

- `package_id` 只表达“风格”
- 新增 `sku_id` 表达“卖什么”
- `/api/generate` 不再直接由前端自由调用
- 必须先有 `paid` 或 `free_granted` 的订单，才能开始生成

---

## 后端需要新增的能力

## 1. 新表

建议至少新增：

- `orders`
- `payment_transactions`
- `refunds`

`orders` 建议字段：

- `order_id`
- `user_id`
- `sku_id`
- `package_id`
- `amount`
- `currency`
- `status`：`created / pending_payment / paid / closed / refunded / failed`
- `provider`
- `provider_trade_no`
- `entitlement_snapshot`
- `created_at`
- `updated_at`
- `paid_at`
- `expired_at`

## 2. 新接口

建议新增：

- `POST /api/orders`
- `GET /api/orders/{order_id}`
- `POST /api/pay/alipay/create`
- `POST /api/pay/wechat/create`
- `POST /api/pay/alipay/notify`
- `POST /api/pay/wechat/notify`
- `POST /api/orders/{order_id}/refund`

## 3. 生成闸门

`POST /api/generate` 前增加强校验：

- 必须带 `order_id`
- 订单必须属于当前 `session_token` 对应用户
- 订单状态必须是 `paid`，或者是免费 SKU 的有效体验单
- 已失效、已退款、已关闭订单不能触发生成

## 4. 回调设计

支付回调要满足：

- 独立公网地址
- 不做登录态校验
- 先验签，再更新订单
- 幂等处理重复通知
- 不能依赖前端跳转结果判断支付成功

---

## 前端需要新增的能力

## 1. 商品层

现在前端只有视觉套餐，没有商品 SKU。

建议新增一个“确认订单”层：

- 免费体验：1 张标清带水印
- 基础套餐：指定风格解锁
- SVIP：阶段性会员权益

视觉套餐页继续负责“选风格”，商品页负责“买什么”。

## 2. 收银台适配

按设备做分流：

- 手机浏览器：展示支付宝 + 微信 H5
- PC 浏览器：展示支付宝网页支付 + 微信扫码支付

## 3. 支付完成回流

支付完成后不要直接认为成功。

前端正确动作应该是：

- 回到站内支付结果页
- 轮询 `GET /api/orders/{order_id}`
- 订单状态变为 `paid` 后，再进入 Waiting 并创建生成任务

---

## 权益模型建议

为了避免后面“支付规则写在 prompt 里”的问题，权益必须后端化。

建议把规则落到 SKU：

- 免费体验：1 张，带水印，标准分辨率，不支持重绘
- 基础套餐：20 张 4K + 2 张 8K，限定部分视觉套餐
- SVIP：全站套餐、更多重绘额度、更多下载规格

后端生成链路只读取订单快照和权益配置，不读取前端文案。

---

## 你这个项目最稳的技术路线

如果目标是尽快上线真实收款，我建议：

### Phase 1

- 支付宝网页支付
- 微信 Native 支付
- 订单表 + 支付回调 + 支付后解锁生成

### Phase 2

- 微信 H5 支付
- 退款接口
- 订单超时关闭
- 支付失败重试

### Phase 3

- 微信 JSAPI
- 会员续费 / SVIP 周期订阅体验优化
- 实体冲印订单

## 风险与注意事项

## 1. 商户主体准备

无论支付宝还是微信支付，真正上线都依赖商户主体、应用/商户号、密钥和平台审核。

如果目前还没有正式商户主体，开发可以先做本地沙箱或假支付联调，但不能直接上线真收款。

## 2. 回调路径与鉴权中间件

当前项目全站 API 主要依赖 `session_token` 鉴权。

支付回调是平台服务器主动调用，不能要求 Bearer Token，所以后续要把支付回调路径加入公共白名单。

## 3. Cloudflare / Nginx / 回调链路

当前线上域名走 Cloudflare → VPS nginx → Docker。

支付回调必须重点验证：

- 回调地址公网可达
- 反代不会拦截 POST body
- 平台请求不会被 WAF 或缓存规则误伤
- 回调超时和重复通知都能正确处理

## 4. 订单与生成解耦

不要让“支付成功”直接同步触发一整条 AI 任务。

更稳妥的做法是：

- 回调只负责改订单状态
- 前端或后端显式调用“开始生成”
- 这样便于重试、补单、退款、风控和客服排查

---

## 最终建议

如果只选一个最现实、最适合当前代码形态的结论，就是：

**先做“支付宝网页支付 + 微信 Native 支付”，把订单模型和支付回调打通；微信 H5 作为紧随其后的补充，JSAPI 放第二阶段。**

这条路径：

- 最符合当前网站形态
- 对现有前后端侵入最小
- 能最快形成“付费解锁生成”的闭环
- 不会把项目过早拖进公众号/小程序体系

---

## 后续开发拆分建议

可以直接拆成 5 个任务：

1. 设计 SKU / 订单 / 权益数据模型
2. 实现订单 API 与 SQLite 表
3. 接支付宝支付创建单与回调
4. 接微信支付创建单与回调
5. 改前端流程：支付成功后再允许生成
