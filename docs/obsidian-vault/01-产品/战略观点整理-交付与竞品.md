# 战略观点整理：前景、OpenClaw、交付定位与竞品

> **文档角色**：思辨、类型学、评审摘要、工程对照——**非拍板真源**。  
> **主指标、MVP 数字、毛利底线、三竞品、成本口径**见 **[[战略决策稿]]**。

---

## 一、项目前景（2026，AI 很热）

- 能成的前提不在「又接了一个大模型」，而在垂直场景（婚纱）有付费理由：情感、仪式感、客单价可覆盖 API 与售后。
- 通用能力会贬值；「去 AI 化体验 + 隐藏参数 + 高奢叙事」是在和聊天框生图做差异化。
- 难成之处：供给暴增；真正决定生死的是交付确定性（十张里九张不崩）和获客，而不是单次神图。
- 关键要素（按重要性粗排）：成片质量可重复；信任与隐私（授权、留存、可验证）；获客与转化路径；单位经济（套餐价 vs 模型与售后成本）。
- **隐私的定位**：隐私与合规更接近**高信任订单的准入门槛**——不直接带来增长；未达标会挡掉愿意付高价、敢把脸交给你的用户。它是**门槛**，不是增长引擎。

- 检索工具某次关键词组合可能返回空结果，不等于产品不存在。
- **OpenClaw** 指 GitHub 上的 [openclaw/openclaw](https://github.com/openclaw/openclaw)：个人 AI 助手、跨端/自托管方向（具体能力以官方文档为准）。

## 三、若「人人都有 OpenClaw / 人人都能自己生成」

- 被削薄的是「能力垄断」，不是全部需求。
- 仍会向这些需求收敛：
  - **确定性**：失败率、售后、规则。
  - **品味与策展**：一套成片而非一张能看。
  - **时间与心力**：不愿再学提示词与管线。
  - **信任与合规**：脸、肖像、链路可解释。
  - **实体与仪式**：相册、请柬等。
  - **身份与社交**：「我们选的」与品牌背书。
- 类比：人人能做菜不等于餐厅消失；工具普及后，卖的是低风险的成套交付，而非多一个生图入口。

## 四、「不是多了一个生图入口」——当前系统是否具备

### 已具备的「非入口」成分

- Director 编排；套餐多图 + 变体提示；多参考图生成。
- VLM 质检 + 多轮；物理问题 Nano 修、情绪问题 GPT Image 修，低分可重生。
- 定时清理上传/成片目录。

### 尚未达到「婚礼级可对外承诺交付」的缺口

- 生成任务主要在内存，与已有 SQLite `tasks` 表脱节，重启/多实例不可靠。
- 质检异常或解析失败时倾向默认「通过」，与「敢说可交付」矛盾。
- 双路径修复失败仅日志，缺少「算不算一张有效成片、占不占额度」的商业语义。
- 跨多张的身份/双人一致性主要靠参考图与 prompt，缺少可测机制。
- 静态文件路径与鉴权策略和「面部数据、全站鉴权」红线未完全对齐。
- 订单、支付、修片轮次、人工升级等服务层仍缺。

### 结论

- 具备「流水线交付」雏形，尚不足以称「我们替你担结果」的签约级服务。

## 五、重点突破口（按投入产出）

1. **订单级生成**：任务与结果持久化、每张元数据（分数、修复/重生轨迹）、失败与不交付的明确语义（含质检不可用时的行为）。
2. **文件与隐私**：避免可猜 URL 直出；按会话/签名/后端校验输出，与「托管责任」叙事一致。（同上：主要为承接高信任订单，别指望单靠「我们更安全」拉新。）
3. **一致性可验证**：短期收紧参考图与 session 内策略；中期用小评测集把通过率/重修率变成指标。
4. **商业规则写入产品与后端**（可并行稍晚）：轮次、张数、超时、部分成功与退款等由**订单/状态机/策略服务**承载；**模型只读取后端组装的上下文并执行推理**，不把定价与额度写进 prompt 当唯一真源。

若只能先做一件事：**持久化 + 失败/不交付语义**，性价比最高。

## 六、与市面「最新 AI 产品」对比——有没有优势

### 对手类型（类型学仍有参考价值）

- **通用大模型生图**：能力强、不负责成套交付。
- **垂直一键工具/App**：快、模板多、浅管线。
- **个人/开源工作流**：灵活、无产品化承诺。

### 购买决策维度（比「你是哪类产品」更狠）

用户最后比的往往不是「通用模型 vs 垂直工具」，而是**结果与风险**：多少钱、多久出片、像不像本人、成套感如何、失败怎么办、朋友圈/请柬场景够不够「炸」。竞品分析应优先按**用户结果**对齐，类型标签只做辅助。

**建议竞品对比表列（待填具体产品）：**

| 维度 | 说明 |
|------|------|
| 像不像本人 | 身份一致性、双人合影可信度 |
| 成套感 | 同一订单内风格/人物连贯性 |
| 出片时间 | 首版、修订后到手时间预期 |
| 失败补救 | 重试、退款、人工兜底、是否写清规则 |
| 隐私责任 | 留存、访问控制、协议能否兑现 |
| 价格带 | 单次/套餐/订阅 |
| 是否人工兜底 | 有无升级路径 |

### 潜在优势

- 管线深度（编排 + 多参考 + 质检 + 分路径修复）——**见下条边界条件**。
- 垂直动线（上传—试妆—套餐—等待—成片）。
- 品牌与审美位。
- 自建 + 数据政策若做实则叙事更强。

### 管线深度何时算优势（边界条件）

管线深**不天然**是优势；仅当它能**显著提高稳定交付率**（或可量化降低废片率、重修成本、客服成本）时才是。否则容易变成更贵、更慢、更难维护——必须用指标证明「深」值得。

### 劣势或不确定

- 分发与信任资产。
- 用户只要「够发朋友圈」时浅工具可能够。
- 模型新不新不构成长期护城河。
- 工程未完成项会削弱隐私与专业叙事。

### 总判

- 优势在「垂直成套工作流 + 质检修复 + 品牌动线」这一方向，非已兑现的市场地位。
- 若定义成「可签约交付」则方向有优势，需把订单、安全取图、失败规则、一致性做实，优势才从概念落到产品。

## 七、「可签约的婚纱级交付系统」指什么

- **含义**：能像影楼接单一样写进条款——交付什么、何时算交片、交不了怎么办、数据如何处理、争议如何举证——而非仅 Demo。
- **通常包含**：
  - 交付物边界（张数、规格、套餐、轮次）。
  - 可执行的验收标准（自动阈值 + 人工规则 + 可追溯）。
  - 失败与例外规则（占不占额度、重试、人工、退款）。
  - 时间与可用性承诺（SLA）。
  - 隐私与数据与协议一致且技术上可兑现。
  - 全流程可追溯供客服与争议使用。
- **与「多一个生图入口」的差别**：入口保证「能生成」；签约级系统保证「在约定条件下，结果与责任边界对用户可预期」。

## 九、工程对照清单（便于拆任务）

与当前代码库的对应关系（随重构更新本表）：

| 突破口 | 现状线索 | 建议方向 |
|--------|----------|----------|
| 订单级持久化 | `generate.py` 中 `_tasks` 内存字典；`database.py` 已有 `tasks` 表但未贯通生成流 | 创建任务写库、状态轮询读库；后台 worker 与任务表一致；可选 Redis/队列后再扩多实例 |
| 每张成片元数据 | 成片仅存文件路径；无「轮次分数、修复路径」结构化记录 | 新表 `task_photos` 或在 `tasks` 上 JSON 字段记录每张照片的 score、fix_rounds、regenerated 等 |
| 质检不可用语义 | `vlm_checker.py` 异常时 `passed=True` 放行 | 产品决策：`quasi_pass` / `blocked` / 触发人工；禁止静默当「优秀交付」 |
| 修复失败语义 | `_dual_path_fix` 失败仅 `logger.warning` | 定义 `photo_status`：delivered / failed / needs_human；与套餐张数、重拍额度挂钩 |
| 文件鉴权 | `middleware/auth.py` 将 `/api/files/` 列为公开；静态挂载直出 | 改为经后端校验 session 后流式返回，或短期签名 URL |
| 一致性评测 | 无固定基准集 | 选 2～3 组参考图 × N 套餐，脚本跑管线统计通过率/重修率，回归对比 |

## 十、下一步可执行项（优先级建议）

1. **P0**：生成任务与状态写入 SQLite（或等价持久化），前端轮询与重启后一致；与 `tasks` 表 schema 对齐。
2. **P0**：文件访问收紧——至少 outputs 需带鉴权或签名；与隐私叙事、红线文档一致。
3. **P1**：质检/解析失败时的显式状态与用户可见说明（不占「满意交付」额度或进入待处理）。
4. **P1**：单张照片结构化状态 + 失败不计入可交付张数的规则（与 [[战略决策稿]]、[[商业模式]] 中重拍/套餐对齐）。
5. **P2**：小型离线评测集 + 每次改 prompt/阈值跑对比报告（可记在 [[07-日志]]）。
