# 妆造预览与状态恢复

## 背景

本轮处理了 3 个直接影响体验和成本的问题：

1. 妆造卡片横向裁切过重，预览里经常看不到完整正脸
2. 用户刷新后，上传档案和妆造流程容易断掉
3. 同一用户反复上传 / 反复进入试妆时，会重复打妆造生成成本

## 本轮决策

### 1. 妆造预览改成“看脸优先”

- 前端不再用 `background-image + cover` 去铺满横卡
- 改成真实 `<img>` 渲染
- 卡片比例改为更接近人像的 `4:5`
- 使用 `object-position: center 18%`，让脸部优先进入可视区域

目的：

- 先保证“能看清人脸”而不是“卡片被铺满”

### 2. 上传状态改为可恢复

- workflow 新增 `uploadedFiles`
- 上传成功后，把每个坑位对应的服务端文件 URL 落到本地 workflow
- Upload 页刷新时，从 `uploadedFiles` 反推页面预览，不再直接回到空态
- 如果用户只是刷新、没有替换任何图片，再次继续时直接沿用当前流程，不清空妆造/套餐状态

说明：

- 这次解决的是“已上传后刷新丢状态”
- “上传前本地暂存草稿”还没做；如果后面要补，需要上 IndexedDB，而不是把大图塞 localStorage

### 3. 后端增加去重与缓存

- 上传同一 `slot` 时，先删除该坑位旧图，再保存新图
- 妆造预览按 `user_id + gender + reference_signature` 做缓存
- 参考图没变时，`/api/makeup/generate` 直接复用已有结果，不再重复调用模型

目的：

- 避免一个用户越试越多历史图堆在目录里
- 避免刷新 / 回退 / 重进页面反复消耗试妆生成

## 当前结果

- 妆造卡片展示更接近人像浏览逻辑
- 已上传用户刷新后可恢复上传预览和流程上下文
- 同坑位重复上传不再累计历史脏数据
- 妆造试妆具备服务端缓存，重复进入不再重复打模型

## 2026-04-02 补充：刷新可见不等于“只加后端缓存”

这轮又暴露出一个很典型的问题：

- 后端已经有妆造缓存
- 但用户在 VPS 上刷新页面，仍然看不到新的预览效果

根因不是单点，而是两层缓存叠在一起：

### 1. 前端 workflow 缓存会拦截重新请求

- `makeupOptions` 会持久化到 `localStorage`
- 如果页面逻辑是“本地已有图就不再请求 `/api/makeup/generate`”
- 那么刷新后看到的只是上一次本地状态，不是后端当前缓存结果

这意味着：

- “服务端已经缓存” != “刷新后用户一定能看到最新效果”

本轮修正策略：

- 页面进入 `/makeup` 后仍然会后台重验一次
- 如果本地已有妆造图，先展示旧图避免空白
- 等后端返回后再静默替换

这样兼顾两点：

- 首屏不闪空
- 刷新后仍能看到线上实际结果变化

### 2. 浏览器图片缓存会掩盖视觉改动

即使前端重新请求了 `/api/makeup/generate`，如果返回的还是同一个私有图片 URL，浏览器也可能继续复用旧图。

本轮修正策略：

- 妆造预览图 URL 增加 `?v=<mtime_ns>` 版本参数
- `/api/files/*` 返回 `Cache-Control: private, no-store, max-age=0`

目的：

- 避免浏览器强缓存让用户误以为“线上没生效”

### 3. 人像��览卡片不应该再用 cover 思路

之前妆造卡片为了铺满卡片，使用了偏展示型的裁切方案：

- `background-image + cover`
- 或 `<img>` 上继续保留类似的裁切策略

问题在于试妆预览的任务不是“做情绪化海报”，而是“看清脸”。

本轮修正策略：

- 改为真实 `<img>` 渲染
- 卡片保持 `4:5`
- 使用 `object-fit: contain`

结论：

- 妆造页的预览优先级应该是“完整可辨识”高于“卡片视觉铺满”

## 验证口径补充

以后凡是用户反馈“刷新后还是旧图 / 看不到变化”，排查顺序固定为：

1. 页面刷新后有没有重新请求 `/api/makeup/generate`
2. 返回的图片 URL 是否带版本参数
3. `/api/files/*` 的响应头是否禁用了浏览器强缓存
4. 前端最终渲染是否仍在用裁切型展示策略

只有这四项都通了，才能认为“刷新可见”真正成立。

3. 给上传和试妆都加频控 / 冷却时间，防止恶意连续点击
