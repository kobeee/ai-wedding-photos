## AI 管线架构（v0.3.0 上下文工程重构）

> 详细理念见 [[上下文工程]]，工程规格见 [[上下文工程-工程落地版]]

```
用户上传照片
    ↓
参考图筛选（reference_selector）
    1主 + 2辅，��量评分排序
    ↓
Creative Brief 获取（briefs.py）
    10 套人工精调模板，叙事 + 弱控制字段
    ↓
Director 编辑模式（可选，仅在有用户偏好时）
    只编辑白名单字段，保护核心叙事
    ↓
Prompt 5层组装（prompt_assembler）
    System Identity → Identity Anchor → Brief → Slots → Variant + Avoid
    ↓
Nano Banana Pro（Gemini）
    参考图 + Prompt → 4K 底图
    ↓
VLM Checker（gemini-3.1-pro-preview vision）
    三维评分：identity_match + brief_alignment + aesthetic_score
    ↓
thresholds 决策引擎
    ├─ identity < 0.70 → 重生（脸不像就别修了）
    ├─ hard_fail + avg 极低 → 重生
    ├─ hard_fail + avg 可救 → 局部修复
    ├─ all pass (avg ≥ 0.80) → 交付
    └─ final delivery floor (0.72) → 低于此值直接拦截
```

### 上下文 5 层架构

| 层 | 职责 | 来源 |
|---|---|---|
| System Identity | 模型角色定义 | 静态常量 |
| Identity Anchor | 参考图定位 | reference_selector |
| Creative Brief | 创意叙事 + 弱控制 | briefs.py + Director |
| Dynamic Slots | 妆造/性别/偏好 | slot_renderer |
| Variant + Avoid | 单张差异化 + 约束 | variant_planner |

### Director 编辑模式（v0.3.0）

不再从零生成 CameraSchema，改为微调 CreativeBrief 边缘字段：

| 情况 | 是否调用 LLM | 理由 |
|------|------------|------|
| 无用户偏好 | 否 | 直接用基础 brief |
| 只有妆造选择 | 否 | 机械插槽替换 |
| 有自然语言偏好 | 是 | 融入 brief |
| 渠道不可用 | 否 | fallback 基础 brief |

### VLM 质检（v0.3.0）

从单一 score 升级为三维评分 + hard/soft fail：

| 维度 | 说明 | 阈值 |
|------|------|------|
| `identity_match` | 人脸身份一致性 | < 0.70 → 重生 |
| `brief_alignment` | 创意意图匹配度 | 纳入 avg 计算 |
| `aesthetic_score` | 美学质量 | 纳入 avg 计算 |
| `hard_fail` | 畸形/穿模等硬伤 | true → 局部修或重生 |

### 2026-03-24 生产语义修正

- `inspection_unavailable` 不再走“本地高分 fallback”
- VLM 不可用时，任务直接 `failed`
- 最终交付前新增 `meets_delivery_floor()` 闸门
- 即使前面某轮逻辑漏掉，保存结果前仍会再次阻断 `score < 0.72` 的图片

### Director 编辑鲁棒性修正

- 在线 `Director` 偶发返回截断 JSON
- 当前实现改为三层兜底：
  - 先正常解析
  - 再从 code fence / 噪音文本提取 JSON
  - 最后从残缺 key/value 中做部分救回
- 若仍失败，再回退基础 brief

这意味着：

- 用户偏好不再因为一次半截 JSON 就完全丢失
- 但外部模型若持续输出严重残缺内容，仍可能部分回退

## 数据层

### SQLite 表结构

```sql
-- 用户表
CREATE TABLE users (
    user_id TEXT PRIMARY KEY,
    session_token TEXT UNIQUE NOT NULL,
    photos_count INTEGER DEFAULT 0,
    created_at TEXT DEFAULT (datetime('now'))
);

-- 任务表
CREATE TABLE tasks (
    task_id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    package_id TEXT DEFAULT '',
    status TEXT DEFAULT 'pending',
    progress INTEGER DEFAULT 0,
    quality_score REAL DEFAULT 0.0,
    message TEXT DEFAULT '',
    result_urls TEXT DEFAULT '[]',  -- JSON array
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
```

数据库文件：`./data/lumiere.db`，Docker volume `backend-data` 持久化。

## 鉴权机制

Session Token Bearer 中间件（`middleware/auth.py`）：

1. 用户上传照片时自动创建 user + 生成 `lst_` 前缀的 token
2. 后续请求通过 `Authorization: Bearer lst_xxx` 鉴权
3. 公开路径白名单：`/api/health`, `/api/upload`, `/api/packages`, `/docs`, 静态文件

Phase 2 升级 JWT。

## ACP Agent 层

三个 Agent 通过 acp-sdk 注册，独立容器端口 8001：

| Agent | 能力 | 用途 |
|-------|------|------|
| `wedding-photographer` | text-to-image, image-to-image, quality-check | 完整拍摄流程 |
| `makeup-artist` | - | 3 种妆造风格预览 |
| `quality-inspector` | - | VLM 质检报告 |

ACP 端点：`GET /agents`（发现）、`POST /runs`（执行）、`GET /runs/{id}`（状态）

> 关键：入口必须是 `acp/__main__.py`，不能直接跑 `acp/server.py`，否则会触发 Python `__main__` 双重导入问题。详见 [[踩坑记录#Python __main__ 双重导入]]。

## 2026-03-24 线上验证结论

- 正确的生产 Compose project name 是 `wedding-photos`
- 误起的 `docker-*` 栈会造成“代码已更新但线上还是旧行为”的假象
- 当前真实生产验证结果是：
  - 上传正常
  - 生成正常启动
  - 若 VLM 正常，则继续产出
  - 若 VLM 超时，则明确失败并拒绝交付

这比之前的“低分也 completed”更符合产品定位。
