### 妆造页“后端已缓存”但刷新后仍看到旧预览
- **日期**: 2026-04-02
- **分类**: 前端 / 后端 / 缓存
- **现象**: `/makeup` 页已经接入妆造缓存，但用户在 VPS 上刷新页面后，仍看不到新的预览效果变化
- **原因**:
  1. 前端把 `makeupOptions` 持久化到 `localStorage`
  2. 页面逻辑此前是“已有本地妆造图就不再请求 `/api/makeup/generate`”
  3. 即使后端已有新缓存结果，刷新后页面也只会显示旧本地状态
  4. 私有图片 URL 不带版本参数时，浏览器还可能继续复用旧图片缓存
  5. 妆造卡片此前使用裁切型展示策略，视觉上更容易让人误判“图没有变化”
- **解决**:
  1. `/makeup` 页面改为进入后总会后台重验一次接口
  2. 若��地已有预览，先展示旧图，再静默替换为后端返回结果
  3. 妆造图片 URL 增加 `?v=<mtime_ns>` 版本参数
  4. `/api/files/*` 响应增加 `Cache-Control: private, no-store, max-age=0`
  5. 卡片渲染改为 `<img> + object-fit: contain`
- **教训**:
  - “服务端缓存生效”不等于“用户刷新可见”
  - 线上图片问题必须同时检查三层：页面状态缓存、接口结果缓存、浏览器文件缓存
  - 试妆预览的核心目标是“看清脸”，不是“把卡片铺满”

## 部署相关

### 点击「AI 试妆」时报 413，实际卡在上传接口总请求体
- **日期**: 2026-04-01
- **分类**: 前端 / 部署
- **现象**: 用户上传 5 张照片后点击进入试妆，前端只显示 `Request failed with status 413`
- **原因**:
  1. 进入 `/makeup` 前，前端会先把 5 张原图一次性 `POST /api/upload`
  2. 前端 nginx `client_max_body_size` 只有 `20M`
  3. 产品允许“原图直传”，单张图虽然没超过后端 `10MB` 限制，但 5 张合计很容易超过 `20M`
  4. nginx 直接拦截后返回 HTML 版 `413`，前端无法从 JSON 里读到可解释文案
- **解决**:
  1. 将 `docker/nginx-frontend.conf` 的 `client_max_body_size` 提升到 `64M`
  2. 前端 `api.ts` 对 `413` 增加明确提示，避免只显示通用状态码
- **教训**:
  - 上传链路要同时看“单文件限制”和“整次 multipart 请求限制”
  - 反向代理的 body limit 必须和产品上传策略一起设计，不能只按接口字段想当然

### 用户真实原图继续撞 413，单纯放大 nginx 限制不够
- **日期**: 2026-04-01
- **分类**: 前端 / 部署
- **现象**: 将 nginx 请求体上限从 `20M` 提到 `64M` 后，用户仍可能因为 5 张手机原图过大而继续触发 `413`
- **原因**:
  1. 现有上传页会把 5 张图一次性提交
  2. 手机原图常见单张 8MB~20MB，5 张一起发很容易继续逼近代理层上限
  3. 问题核心不是“用户不会压图”，而是前端没有自动处理大图
- **解决**:
  1. 上传页提交前自动将图片收敛到长边 `2200`、目标体积 `3MB` 内
  2. 底部按钮增加“正在优化照片...”状态，避免用户误以为卡死
  3. nginx 上限继续提高到 `96M`，给极端场景留缓冲
- **教训**:
  - 对面向真实手机原图的产品，自动上传优化不是锦上添花，是基础能力
  - 不能把“请用户自己压缩”当成默认交互

### 批量上传失败时，不能把“上传报错”直接等同于“试妆链路不可用”
- **日期**: 2026-04-01
- **分类**: 前端 / 后端 / 部署
- **现象**: 用户在“下一步：AI 试妆”阶段看到上传报错，很容易被误判成“试妆接口坏了”或“线上整体不可用”
- **原因**:
  1. Upload 页在进入 `/makeup` 前要先完成 `/api/upload`
  2. 批量上传失败，前端此前会直接中断，不会继续尝试更细粒度的上传
  3. 真实线上还存在另一类失败：
     - 并非 `413`
     - 而是 5 张图一起提交时被 AI 预检按身份一致性直接拒绝
  4. 如果不拆开验证，很容易把“上传前置失败”和“试妆生成失败”混为一谈
- **解决**:
  1. 前端对批量上传 `413` 增加自动 fallback：
     - 改为按 `slot` 逐张上传
  2. 后端 `/api/upload` 兼容单张带 `slot` 上传：
     - 只在一次请求里收齐必传坑位时才做整包校验
  3. VPS 真实验证时分两步验：
     - 先看批量上传为什么失败
     - 再用同一 session 验证逐张上传后能否继续 `AI试妆`
- **验证结果**:
  1. VPS 上批量 5 张上传当前返回 `400`：
     - `两张新郎照片不像同一个人，请替换其中一张`
  2. 同一 session 下逐张上传 `couple_full / groom_portrait / groom_full / bride_portrait / bride_full` 全部 `200`
  3. 随后男/女 `/api/makeup/generate` 都返回 `200`
  4. 生成出的试妆图 URL 在带 cookie 访问时可正常返回 `200`
- **教训**:
  - “上传阶段失败”不等于“试妆服务挂了”
  - 上传链路必须具备从批量到逐张的自动降级能力
  - 线上验收必须把“接口成功”和“结果图实际可读”分开验证

### 前端配图未显示 — Docker 构建缓存 + Compose 项目名冲突
- **日期**: 2026-03-17
- **分类**: 部署
- **现象**: 设计稿中的 5 张配图（Hero 背景 + 4 张视觉套餐卡片）在线上页面不显示
- **原因（多层）**:
  1. 前端代码根本没引用图片 — `gallery__img` 是空 div，Hero 背景是纯色 `#1A1A1A`
  2. 图片文件在项目根目录 `images/` 下，未复制到 `src/frontend/public/images/`
  3. 修复代码后部署，`docker compose build` 使用了构建缓存，新加的 `public/images/` 没进容器
  4. 从不同目录执行 `docker compose` 导致项目名不一致（`wedding-photos-` vs `docker-`），端口冲突
- **解决**:
  1. 将 5 张图片复制到 `src/frontend/public/images/` 并重命名为语义化文件名
  2. Landing.tsx: Hero 加 CSS 背景图，Gallery 卡片加 `backgroundImage` 内联样式
  3. 部署时使用 `docker compose build --no-cache` 强制重建
  4. 统一用 `-p wedding-photos` 固定 Compose 项目名
- **教训**:
  - 设计稿有图片时，前端实现必须同步引用，不能留空占位
  - Docker 部署涉及静态资源变更时，务必 `--no-cache` 重建
  - `docker compose` 始终指定 `-p` 项目名，避免目录名导致的命名混乱

### 已部署到 VPS，但公网仍命中旧页面
- **日期**: 2026-03-19
- **分类**: 部署
- **现象**: 新前端代码、插图资源和容器都已构建完成，但公网首页仍显示旧页面，用户误以为“部署没生效”
- **原因**:
  1. VPS 上残留了旧 project name 的容器：`wedding-photos-frontend-1` / `wedding-photos-backend-1`
  2. 新栈从 `docker/docker-compose.yml` 启动后，project name 变成 `docker`
  3. 公网端口 `3080` 仍被旧前端容器占用，导致流量继续打到旧站
  4. 新容器内其实已经包含新的静态资源和页面代码，但没有接管流量
- **解决**:
  1. 先用 `docker ps` 确认 `3080` 到底绑定在哪个容器上
  2. 手动删除残留旧容器
  3. 在 `/opt/apps/wedding-photos/docker` 下执行 `docker compose -f docker-compose.yml up -d --force-recreate`
  4. 发布后同时验证首页、健康接口和插图 URL
- **教训**:
  - 只看“镜像构建成功”没有意义，必须确认公网流量已切到新容器
  - compose 路径或工作目录变化后，第一反应就是排查历史 project name 残留
  - 发布验收必须包含静态资源 URL，而不是只看首页 `200`

### 以为“代码修完”就算完成，实际任务目标是 VPS 真试跑
- **日期**: 2026-03-30
- **分类**: 流程 / 部署
- **现象**: 本地修复和本地验证都做完了，但还没有部署到 VPS；用户实际无法开始真实试拍
- **原因**:
  1. 错把“代码完成”当成“任务完成”
  2. 没有把“VPS 真实验证”当成强制收口条件
  3. 汇报时没有明确区分“本地已完成”和“线上已完成”
- **解决**:
  1. 把规则写入项目级 `AGENTS.md`
  2. 以后凡是涉及 `VPS/线上验证/真实试跑/上线联调` 的任务：
     - 本地 build/lint 只算前置检查
     - 必须部署到线上
     - 必须至少跑一条真实主链路
- **教训**:
  - 对这个项目，唯一有效的验收是“线上主链路已经跑通”
  - 如果用户目标是拿真实照片开拍，本地通过没有任何交付价值

### session 过期字段存在，但实际没有生效
- **日期**: 2026-03-30
- **分类**: 后端 / 安全
- **现象**: `sessions.expires_at` 已写入数据库，但过期 session 仍然可以继续访问订单与图片
- **原因**:
  1. 登录态校验时只按 `session_token` 查记录
  2. 查询条件里没有检查 `expires_at > now`
- **解决**:
  1. `get_user_by_token()` 增加过期条件
  2. 让 session 过期真正成为鉴权边界
- **教训**:
  - “写入过期时间”不等于“真正有过期能力”
  - 任何安全字段都必须验证读路径是否真的使用

### 24 小时清理任务误删已交付图片
- **日期**: 2026-03-30
- **分类**: 后端 / 交付
- **现象**: 系统配置里付费单保留 30 天，但清理任务会在 24 小时后直接删除 `outputs/` 下的交付文件
- **原因**:
  1. 文件清理逻辑只看统一的 `data_retention_hours`
  2. 没有区分：
     - 上传原图
     - 已交付成片
  3. 也没有读取订单里的 `retention_days`
- **解决**:
  1. `uploads` 继续按统一小时数清理
  2. 已交付 `outputs` 改为按订单 `retention_days` 计算过期时间
  3. 未绑定交付记录的 `outputs` 仍按默认时效清理
- **教训**:
  - 清理策略不能只看文件目录，必须看业务语义
  - “用户承诺保留多久”必须直接落实到存储生命周期里

### 试妆接口返回部分成功结果，前端可能继续带空卡片流转
- **日期**: 2026-03-30
- **分类**: 前端 / 后端 / AI管线
- **现象**: 某一张试妆图生成失败时，接口仍可能返回长度正确但包含空字符串的数组，前端状态容易进入半残状态
- **原因**:
  1. 后端只检查“是否全空”
  2. 前端只按数组长度判断“是否准备完成”
- **解决**:
  1. 后端要求三张试妆图必须完整
  2. 任一风格图缺失则整体失败
  3. 由前端统一切到样片兜底
- **教训**:
  - AI 结果不能只做“非空兜底”，必须保证结构完整性
  - 前后端对“成功”的定义必须一致

## AI 管线相关

### AI 试妆页面超时降级 — 前端超时太短 + 后端串行调用
- **日期**: 2026-04-01
- **分类**: AI管��� / 前端 / 后端
- **现象**: 进入 AI 试妆页面后转圈很久，最终显示"AI 试妆当前响应较慢，已切换为样片预览，不影响正式生成"
- **原因**:
  1. 后端 `/api/makeup/generate` 对 3 种妆造风格是**串行**调用 Nano Banana 的（`for style in ALL_STYLES`）
  2. 每次 Nano Banana API 调用耗时 20-32 秒（图生图正常耗时）
  3. 3 次串行 = 60-90 秒
  4. 前端 `requestMakeupPreview()` 设了 `AbortController` 超时只有 **20 秒**
  5. 20 秒连第一张图都等不完，必然 abort → `Promise.allSettled` 判定 rejected → 降级到样片
- **排查过程**:
  1. SSH 到 VPS 查看 `docker logs wedding-photos-backend-1`
  2. 从日志时间戳算出每次 API 调用耗时和总耗时
  3. 对照前端超时设置，确认根因是超时不匹配
- **解决**:
  1. **后端改并行**：`asyncio.gather` 并发调用 3 种风格，耗时从 ~90s 降到 ~30s
  2. **前端超时延长**：从 20s 改到 90s，给 API 波动留充分余量
- **教训**:
  - 涉及多次外部 API 调用的接口，第一反应应该检查是串行还是并行
  - 前端超时设置必须基于后端实际耗时来定，不能拍脑袋
  - 日志时间戳是定位耗时问题最直接的工具

### laozhang Nano `imageSize` 契约变更，`1024` 不再可用
- **日期**: 2026-03-27
- **分类**: AI管线 / 后端
- **现象**: `Nano Banana` 线上持续返回 `400 upstream_error`，但请求结构、鉴权和多参考图逻辑看起来都正常
- **原因**:
  1. 旧认知里 `imageSize` 仍按 `256 / 512 / 1024` 使用
  2. laozhang 当前 Nano 合同实际要求的是 `1K / 2K / 4K`
  3. 因此 `imageSize = "1024"` 会被代理层直接拒绝
- **解决**:
  1. 直接在 VPS 上用生产 key 做最小请求探测
  2. 确认 `1024 -> 400`，`1K -> 200`
  3. 后端加入兼容映射：
     - `256 / 512 / 1024 -> 1K`
     - `2048 -> 2K`
     - `4096 -> 4K`
- **教训**:
  - 代理层 API 合同不能只信历史笔记，必须拿生产 key 做最小复现确认
  - 图像生成链路出现 `400` 时，除了查 payload 结构，也要优先核对“枚举值是否已经变更”

### 4K 交付不要直接等同于首轮模型 4K
- **日期**: 2026-03-27
- **分类**: AI管线 / 产品实现
- **现象**: 产品需要“婚纱大片 4K 交付”，但如果把首轮 Nano 全量切到 4K，会明显增加耗时和失败风险
- **原因**:
  1. “生成分辨率”与“交付分辨率”是两个不同层级的问题
  2. 用户要的是下载结果是 4K，不是要求每一次中间生图都原生 4K
- **解决**:
  1. 主生成链路保持稳定分辨率
  2. 只在通过 VLM、准备交付时把正式成片升档到长边 `4096`
  3. 在 Review 页面明确写成“下载当前 4K 大图”
- **教训**:
  - 不要把营销语义直接翻译成首轮模型参数
  - 交付质量应该落在交付层，而不是强行压在整条生成链路上
  - 分辨率验收必须看最终落盘文件尺寸，不是只看页面文案

## ACP / Agent 相关

### acp-sdk uvicorn 兼容
- **日期**: 2026-03-19
- **分类**: 后端 / ACP
- **现象**: `AttributeError: module 'uvicorn.config' has no attribute 'LoopSetupType'`
- **原因**: uvicorn 0.35+ 将 `LoopSetupType` 改名为 `LoopFactoryType`，acp-sdk 1.0.3 还在用旧名
- **解决**: `requirements.txt` 锁定 `uvicorn[standard]==0.34.0`
- **教训**: acp-sdk 是 IBM 维护的，更新节奏不一定跟得上 uvicorn 的 breaking change。涉及第三方 SDK 时锁版本最稳

### Python __main__ 双重导入
- **日期**: 2026-03-19
- **分类**: 后端 / ACP
- **现象**: ACP Server 启动正常，`/agents` 返回空列表，但 `server.agents` 确实有 3 个对象
- **原因**: `python -m acp.server` 把 `acp/server.py` 加载为 `__main__` 模块。agent 文件 `from acp.server import server` 导入的是 `acp.server` 模块（另一个实例）。两个 Server 实例，agent 注册在后者，`run()` 跑的是前者
- **解决**: `acp/server.py` 只做实例化 + import，`acp/__main__.py` 做入口。command 改为 `python -m acp`
- **教训**: Python `-m` 运行会把目标模块加载为 `__main__`，如果其他模块也 import 同一文件会产生两个独立实例。入口逻辑永远放 `__main__.py`

### acp-sdk server.run() asyncio 嵌套
- **日期**: 2026-03-19
- **分类**: 后端 / ACP
- **现象**: `RuntimeError: asyncio.run() cannot be called from a running event loop`
- **原因**: `server.run()` 内部自带 `asyncio.run(self.serve(...))`，不能在 async 函数里调用
- **解决**: `server.run()` 必须在同步上下文中调用

### acp-sdk metadata 参数类型
- **日期**: 2026-03-19
- **分类**: 后端 / ACP
- **现象**: `metadata={"version": "0.1.0"}` 传 dict 不报错但行为不对
- **解决**: 用 `Metadata(tags=[...], capabilities=[Capability(name=..., description=...)])`

### ACP Server 默认绑定 127.0.0.1
- **日期**: 2026-03-19
- **分类**: 部署 / ACP
- **现象**: 容器内 ACP Server 无法从外部访问
- **解决**: `server.run(host="0.0.0.0", port=settings.acp_port)`

## 部署相关（续）

### Docker nginx DNS 解析失败
- **日期**: 2026-03-19
- **分类**: 部署
- **现象**: frontend 容器 crash loop，`host not found in upstream "backend"`
- **原因**: nginx 启动时立即解析 upstream 主机名，Docker 内部 DNS 还没就绪
- **解决**: nginx 配置加 `resolver 127.0.0.11 valid=10s;`，用 `set $backend_upstream` 变量延迟解析
- **教训**: Docker Compose 中 nginx 反代其他容器时，永远用 resolver + 变量方式，不要直接写死 upstream

### Docker 端口残留（port already allocated）
- **日期**: 2026-03-19
- **分类**: 部署
- **现象**: `docker compose down && up -d` 后仍报 `port 3080 already allocated`
- **原因**: docker-proxy 进程没有随容器清理，残留占端口
- **解决**: `fuser -k 3080/tcp` 或 `systemctl restart docker`
- **教训**: `docker compose down` 不一定能完全释放端口，遇到端口冲突先查 `lsof -i :PORT`
