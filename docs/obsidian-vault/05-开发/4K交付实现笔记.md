# 4K交付实现笔记

> 把“婚纱大片应该是 4K”从产品话术落成真实交付能力的研发记录。

---

## 一句话结论

- 当前线上“正式成片下载”已经是 **长边 4096px 的 4K 交付**。
- 实现方式不是强迫 [[Nano-Banana-Pro]] 首轮直接出 4K，而是：
  - 主生成链路保持当前稳定分辨率
  - 仅在通过 [[VLM质检管线]]、准备落盘交付时，再升档到 `4096` 长边

---

## 背景问题

在产品语义里，“婚纱大片”天然对应 4K 甚至 8K。  
但工程上如果把首轮 Nano 生图直接切到 4K，会同时抬高：

- 生成耗时
- 上游失败率
- VPS 任务排队压力
- 重试成本

这次已经验证过，线上链路真正稳定的关键不是一开始就追求最高分辨率，而是先保证：

1. 上传成功
2. 身份锚定命中
3. Nano 正常出图
4. VLM 正常放行
5. 最终结果可稳定交付

所以“4K”应该落在 **交付层**，而不是整个生成过程的每一步。

---

## 设计决策

### 方案选择

采用两段式：

1. **生成阶段**
   - 继续使用当前稳定的 Nano 请求配置
   - 避免把主链路重新变脆

2. **交付阶段**
   - 只有当图片通过 VLM 质检、满足 delivery floor 时
   - 才对最终成片执行升档
   - 统一输出为长边 `4096px`

### 为什么不直接让 Nano 出 4K

- 线上已验证 laozhang 当前 Nano 合同有变动，`imageSize` 需要用 `1K / 2K / 4K`
- 在刚修复完 `imageSize` 契约问题的前提下，再把首轮全部切到 `4K`，风险过高
- 首轮 4K 一旦不稳定，用户拿不到任何图；而交付阶段 4K 只作用于“已经通过质检的图”，风险可控

---

## 这次顺手确认的一个关键事实

这轮排查还顺带确认了 laozhang 当前 Nano 契约与旧文档不一致：

- `imageSize = "1024"` 会返回 `400`
- `imageSize = "1K"` 会返回 `200`

因此当前后端已经做了兼容映射：

- `256 / 512 / 1024 -> 1K`
- `2048 -> 2K`
- `4096 -> 4K`

这部分属于首轮生图稳定性的根因修复，见 [[2026-03-27]]。

- `src/backend/services/delivery_image.py`
  - 新增正式交付图片处理服务
  - 处理逻辑：

    - 长边统一拉到 `4096`

    - 加一层轻量 `UnsharpMask`

- `src/backend/routers/generate.py`

  - 被 VLM 拦截的图片不做升档

### 前端交付语义

- `src/frontend/src/pages/Review.tsx`
  - 下载按钮文案改为：`下载当前 4K 大图`

## 经验总结

### 1. 不要把营销语义直接映射成首轮模型参数

“用户要 4K” 不等于 “模型第一步必须 4K”。  
先把稳定性做对，再把分辨率补在正确的层级。

### 2. 交付分辨率与生成分辨率应该解耦

一个是“模型生成成本问题”，一个是“用户下载体验问题”，不要绑死。

### 3. 验证 4K 不能只看文案

必须同时验证：

- 后端日志是否真的执行了升档
- 最终落盘文件的像素尺寸
- 前端下载入口是否和真实输出语义一致

### 已完成

- 4K 正式交付

### 下一步可做

1. **8K 海报链路**
   - 不建议默认全量执行
   - 更适合做精选成片 / 海报模式单独触发

2. **交付元数据**
   - 可以在结果接口里显式返回：
     - width
     - height
     - delivery_tier

3. **更强的升档策略**
   - 当前是工程内轻量升档
   - 后续如要追求更强海报质感，再考虑独立超分模型或单独付费链路
