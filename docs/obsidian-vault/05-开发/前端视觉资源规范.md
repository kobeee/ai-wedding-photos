# 前端视觉资源规范

> 目标是让 `page.pen` 到前端代码的资源映射可预测、可构建、可上线验证。

---

## 目录规范

前端静态视觉资源统一放在：

```text
src/frontend/public/images/
```

适用范围：

- Hero 背景图
- Landing/Package/Review 等页面插图
- `generated-*.png` 这类设计稿导出的临时命名资源
- 语义化命名的套餐封面图

> [!warning]
> 不要把线上要访问的页面插图继续放在项目根目录 `images/`。那样不会自动进入 Vite 最终构建产物。

---

## 引用规范

页面代码统一使用绝对路径引用：

```ts
'/images/hero-bg.png'
'/images/generated-1773678492426.png'
```

不要使用：

- `../../images/foo.png`
- `public/images/foo.png`
- 依赖运行时目录结构的相对路径

---

## 命名规范

推荐两类命名并存：

- 设计稿追踪阶段：允许保留 `generated-时间戳.png`
- 稳定资产阶段：改为语义化命名，如 `package-onsen.png`

建议：

- 进入长期维护前，把关键视觉图逐步改成语义化文件名
- 临时资源至少要在代码里有稳定引用，不要留无引用孤儿文件

---

## 页面与资源映射

重点页面资源来源：

- `Landing.tsx` / `Landing.css`
- `Makeup.tsx`
- `PackageSelect.tsx`
- `Review.tsx`

校验原则：

- 每个页面引用的资源都能在 `public/images/` 找到
- 每个资源上线后都能通过公网 URL 直接访问

## 发布前校验

### 本地校验

```bash
rg -n "/images/" src/frontend/src
find src/frontend/public/images -maxdepth 1 -type f | sort
cd src/frontend && npm run build
```

### 构建后校验

```bash
find src/frontend/dist/images -maxdepth 1 -type f | sort
```

### 线上校验

```bash
curl -I -s https://wedding.escapemobius.cc/images/generated-1773678492426.png
```

## 常见错误

### 错误 1：设计稿里有图，代码里没有引用

- 结果：页面是空占位或纯色块
- 修复：在组件或 CSS 中显式引用图片

### 错误 2：图片文件存在，但目录不对

- 结果：本地可能看不出问题，构建后线上丢图
- 修复：统一迁移到 `src/frontend/public/images/`

### 错误 3：以为是资源问题，实际是旧容器还在服务

- 结果：容器内已经有新图，但公网仍是旧站
- 修复：回到 [[VPS排障清单]]，先查端口和残留容器

---

## 与视觉还原工作的关系

- `page.pen` 视觉稿校对时，截图对比不只看布局和字距
- 资源缺图会直接导致对比失真
- 因此视觉还原任务的最小前提是：资源目录、资源引用、线上资源可访问性同时成立
