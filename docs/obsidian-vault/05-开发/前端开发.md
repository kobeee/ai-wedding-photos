## 组件开发规范

### 文件组织
- 每个页面：`PageName.tsx` + `PageName.css`
- 公共组件：`components/` 目录
- 一个组件一个文件，不做过度拆分
- 静态视觉资源统一放 `public/images/`

### 状态管理
- Phase 1 使用 React 内置 `useState` / `useContext`
- 不引入 Redux/Zustand（复杂度不够高）
- 页面间数据传递通过 URL params + Context

### 与后端对接
- 通过 `fetch` 调用 [[API设计]] 定义的接口
- 请求路径统一使用 `/api/` 前缀
- Nginx 反代到后端，参见 [[Nginx配置]]

## 2026-03-27 后续前端任务：Upload 真链路 + 合照弱锚定

结合 [[身材比例锚定方案]]，前端后续不再只停留在本地 `useState` 预览，需要补齐真实上传链路。

### 第一阶段：真实上传链路

- Upload 页面调用 `/api/upload`
- 上传完成后持久化 `user_id` 和 `session_token`
- 后续 `/api/makeup`、`/api/generate` 统一透传 `Authorization: Bearer <session_token>`

### 第二阶段：合照弱锚定入口

- Upload 页面改为三段式：
  - 双人合照区
  - 新郎个人照区
  - 新娘个人照区
- 合照区是强提示但不强制
- 文案要明确“建议上传双人全身日常合照，可提升双人照身材还原度”
- 缺少合照时，不要禁用下一步

### 产品边界

- 前端只负责强引导，不负责验证“这一定是有效双人合照”
- 第一版不做前端人数检测
- 前端目标是把 `role=couple / groom / bride` 正确上传给后端

## 2026-03-30 上传页固定坑位升级

这轮把上传体验从“建议 5-10 张”改成了固定坑位清单：

- 必传 4 张：
  - `groom_portrait`
  - `groom_full`
  - `bride_portrait`
  - `bride_full`
- 推荐 1 张：
  - `couple_full`

### 前端改动

- `Upload.tsx` 改为固定坑位状态模型，不再按角色数组堆图
- 每个坑位只允许 1 张，并显式上传 `slot`
- 页面尝试加入参考示意图与取景框：
  - 目标是强调“只看取景范围，不要求同款穿搭”
  - 目标是让用户一眼看懂该传谁、该拍多近、是否需要全身
- 浏览器侧新增上传前压缩：
  - 最大长边约 `2200px`
  - 统一转为 JPEG
  - 目标大小约 `2.6MB`

### 目的

- 直接解决“张数和要求太模糊”的问题
- 降低原图直传导致的总请求体积波动
- 缓解前端侧 `request body too large` 类上传报错
- 保持上传页视觉调性，不把它做成低质表单

### 截至 2026-03-30 深夜的真实状态

- 用户在线上 VPS 截图中，仍然看到抽象占位块
- 因此“参考示意图已经在用户侧生效”不能算完成
- 当前更准确的表述应是：
  - 前端代码已经尝试替换这块示意
  - 但用户验收结果表明，线上实际生效资源仍未确认一致

后续若继续排查，优先级应是：

- 先确认用户看到的 `/upload` 实际加载了哪份 CSS / 静态资源
- 再确认是否存在 CDN / 浏览器缓存
- 最后才是继续调整示意图本身的视觉设计

## 2026-03-30 深夜补记：上传页示意图改真人样片，撤销浏览器预压

这轮根据用户线上反馈，重新定性了两个问题：

- 问题一不是“缓存没刷新”，而是当前上传页确实在用 `upload-guides/*.svg` 线稿
- 问题二是浏览器侧上传前压缩会直接丢失用户原图上限，这不该成为默认策略

### 前端处理

- 上传页示意图改为项目内已有样片裁切版 `upload-guides/*.webp`
- 保留取景框和标签，但底图不再是抽象线稿
- 上传页文案整体压短：
  - 标题区不再重复解释
  - 坑位说明改成短句
  - 底部 checklist 改成三条极短提示

### 链路策略修正

- 移除浏览器侧上传前压缩
- 前端选择文件后直接保留原文件对象上传
- 上传卡片文案同步改为：
  - `原图直传，服务端保留原始文件`

### 当前结论

- “为了躲 payload 限制而在浏览器先改码” 这个方向不成立
- 正确分层应该是：
  - 上传与存储层保留原图
  - 调模型前再做预算化派生压缩

## 2026-03-31 当前未闭合问题

根据最新验收反馈，上传页这轮还不能算完成，原因不是单点 bug，而是产品、设计、素材三层同时没有闭环。

### Upload 示意图问题

- 当前示意图虽然已经不是抽象线稿，但依然不符合坑位目标
- 具体表现：
  - 双人合照看不到完整全身
  - 半身图看不到脸
  - 全身图主体不明确

这说明：

- 不能只做“替换成真人图”
- 必须按坑位重新设计示意图素材和裁切规则

### 固定 5 张策略待重审

- 当前前端已经实现为：
  - 4 张必传
  - 1 张推荐
- 但这是不是正确的最终产品策略，当前并没有被验证

后续必须重新判断：

- 5 张是不是足够
- 是否应该存在“最低要求”和“最佳效果”两套口径
- 固定坑位是否写死得过早

### 视觉素材人种约束

- 当前样片与示意图里存在明显偏欧洲人的人物
- 产品侧要求应明确写成：
  - 模特必须是中国人
  - 不接受“东亚即可”的模糊表达

这条约束后续应同步影响：

- Upload 示意图
- 套餐样片
- 试妆兜底图
- 视觉稿素材

### `page.pen` 已落后

- 当前前端 Upload 页已经和 `page.pen` 不一致
- 若后续继续直接改代码，设计真源会进一步失效

因此后续顺序应调整为：

1. 先回改 `page.pen`
2. 再以视觉稿为准推进前端 UI
