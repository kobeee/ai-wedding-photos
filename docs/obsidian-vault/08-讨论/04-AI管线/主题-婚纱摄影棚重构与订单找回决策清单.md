# 主题-婚纱摄影棚重构与订单找回决策清单

## 本轮排查结论

### 1. “选不同套餐却总是看到 699”不是单点 bug，而是产品结构表达失败

当前实现里：

- `package_id` 是视觉主题
- `sku_id` 是销售权益包
- 结账页默认选中主推 SKU `memory_699`

对应代码：

- `src/frontend/src/pages/Checkout.tsx`
- `src/backend/models/database.py`
- `docs/obsidian-vault/08-讨论/02-架构/主题-匿名账户与订单售后体系讨论.md`

因此用户感知到的不是“我选错了”，而是：

- 前一步在选“拍什么风格”
- 后一步突然变成“买哪档权益”
- 页面文案虽然解释了 `package` / `sku` 的区别，但用户仍然把它们都理解成“套餐”

### 2. `scene_count` 目前只是权益数字，不是实际可选内容

当前前端只允许选择一个 `scene chip` 来辅助推荐视觉主题，没有把多景选择写入订单。

因此：

- `399 / 699 / 999` 上的 `3景 / 5景 / 7景`
- 与用户实际下单时“选了哪些景”
- 与后端最终生成哪些变体

目前没有真正连起来。

### 3. 订单其实已经在后端保存了，但前端没有做“找回入口”

当前后端已有：

- `identity`
- `session`
- `orders`
- `generation_batches`
- `deliverables`

并且同一 cookie session 下已支持：

- 创建订单
- 查询订单列表
- 查询订单详情
- 查询交付结果

缺的是：

- 前端“我的订单”入口
- 显性展示订单号
- 跨设备找回机制
- 支付后轻绑定能力

### 4. 当前 AI 管线仍然是“大一统单次生成 + 质检修补”模式

现状不是完全没有分层，而是分层还不够细：

- 参考图筛选：`reference_selector.py`
- prompt 组装：`prompt_assembler.py`
- variant 规划：`variant_planner.py`
- 主生成：`nano_banana.multi_reference_generate`
- 事后质检：`vlm_checker`
- 失败后修补：`_dual_path_fix()`

但核心问题是：

- 仍然让首轮大模型一次性同时解决身份、构图、服饰、场景、动作、情绪
- 修补阶段只能“在已经生成出的图上补”
- 没有把“脸、比例、服饰、场景、动作”拆成独立可控的中间资产

## 我建议的重构方向

参考 Harness 类工程的思想，目标不是“再换一版 prompt”，而是把婚纱摄影生产线拆成多阶段可观测管线：

1. 身份锚定层
   - 提取新郎/新娘正脸锚点
   - 提取双人比例锚点
   - 给每个锚点独立评分

2. 结构规划层
   - 先决定本单要交付哪些景别、构图、镜头、动作
   - 明确每一张图的角色分工，而不是直接循环 `variant`

3. 场景/服饰生成层
   - 先把场景、光线、服饰、动作做成稳定底图
   - 这一层不追求最终脸准

4. 身份融合层
   - 在后段再做精确身份保持/替脸/局部重绘
   - 把“脸像不像本人”从首轮生图压力里拆出来

5. 质检与返工编排层
   - 不是统一 repair
   - 而是根据失败类别走不同返工路径

## 需要用户先拍板的决策

### A. 产品结构是否继续保留 `package` / `sku` 二层

选项含义：

- 保留二层：继续风格和权益分离，但前端必须重写命名和页面结构
- 合并成一层：直接卖“699 冰岛 / 699 极简影棚 / 699 中式庭院”这样的商品

这是最先要定的，因为它直接决定前端信息架构和订单模型要不要改。

### B. 多景权益到底怎么兑现

必须先定一种：

- 用户下单时显式选择 3/5/7 个场景
- 系统按已选视觉主题自动扩展出 3/5/7 个子景别
- 混合模式：先选主主题，再补若干子场景

不定这个，`scene_count` 永远只是一个展示数字。

### C. 订单找回优先走哪条路

建议只选一个主路径先落地：

- 同设备 cookie session + 订单页入口
- 邮箱轻绑定
- 支付后短信/邮箱凭证

我的建议顺序是：

1. 先补“同设备可见订单号 + 我的订单列表”
2. 再补邮箱轻绑定

### D. 是否接受“匿名体验 + 支付后才绑定邮箱”的策略

这是当前最稳的折中：

- 首屏不登录
- 上传到下单保持匿名 session
- 付款后提示留邮箱，用于订单找回与交付通知

如果你坚持“生成前就收邮箱”，整个漏斗体验会变化。

### E. AI 管线第一阶段要做到什么程度

建议分两档：

- V1.5：先做“弱拆分”
  - 继续用现有模型栈
  - 增加 shot plan、scene plan、identity anchor scoring、失败分类返工
- V2：做“强拆分”
  - 引入明确的底图生成 + 身份融合/替脸阶段
  - 允许多阶段中间产物落盘与回放

如果直接上 V2，周期更长，但收益也更大。

### F. “最后一步替脸”是否是可接受的产品策略

这是最关键的产品和伦理决策之一。

因为一旦明确采用：

- “先生成整张图”
- “最后再做人脸身份融合”

那我们在工程上就会主动围绕这个假设设计中间件、质检和返工逻辑。

如果你不接受这个方向，就应该走“从首轮生成开始就强身份锚定”的路线。

## 我当前建议

如果要兼顾速度和结果，我建议按下面顺序推进：

1. 先修前端表达问题
   - 让用户清楚区分“风格主题”和“交付档位”
   - 把订单号和订单列表做出来

2. 把 `scene_count` 从展示数字变成真实订单数据
   - 允许订单记录“本单选了哪些景”

3. AI 管线先做 V1.5 弱拆分
   - shot plan
   - anchor scoring
   - failure routing

4. 你确认“末段身份融合/替脸”可接受后，再进 V2

这样风险最可控，也最符合当前工程基础。
