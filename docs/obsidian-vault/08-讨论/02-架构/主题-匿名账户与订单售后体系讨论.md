# 主题-匿名账户与订单售后体系讨论

> 围绕“尽量不让用户登录”的前提下，如何把匿名身份、支付、交付、重拍与售后串成一条可追踪主线的讨论纪要。

## 讨论背景

当前站点已经具备匿名 session 雏形：

- 上传时创建 `user_id + session_token`
- 后续试妆、生成、审片依赖该 token
- 前端本地持久化工作流状态

这套机制适合试玩，但当产品进入真实收费后，还需要承接：

- 订单支付
- 权益解锁
- 交付张数承诺
- 重拍与微调额度
- 售后找回与退款

## 本轮核心结论

### 1. 不强制登录是对的，但不能没有身份层

方向应为：

**默认匿名账户 + 服务端会话 + 可选轻绑定**

绑定只在真正需要时触发，例如：

- 跨设备查看
- 支付后找回
- 延长保留
- 售后处理

### 2. 真正的业务主键应从 `task` 切换到 `order`

收费后卖的不是一次生成调用，而是一份履约承诺，包括：

- promised photos
- 分辨率规格
- 重拍/重绘额度
- 保存周期
- 不足补偿

因此主线应改为：

**匿名身份 -> 订单 -> 支付 -> 权益生效 -> 生成批次 -> 交付清单 -> 售后工单**

### 3. `package_id` 和 `sku_id` 必须拆开

- `package_id` 表示视觉风格/拍摄主题
- `sku_id` 表示销售商品/权益包

不能继续把风格选择与商品购买混为一个“套餐”概念。

### 4. 支付成功不等于立刻开始生成

支付回调只做：

- 验签
- 幂等更新支付状态
- 激活订单权益

真正开始生成应是显式履约动作，便于：

- 重试
- 补单
- 风控
- 客服排查

### 5. 售后应进入正式状态机

至少要区分：

- 技术售后
- 审美售后
- 履约售后
- 退款售后

并明确哪些场景：

- 不扣用户额度
- 消耗重拍/微调额度
- 需要人工介入
- 进入部分退款或全额退款

## 本轮识别出的风险

### 1. 匿名账户边界还不够稳

当前更接近“上传即开户，token 放前端本地”。

对试玩可用，对付费订单不够稳。后续应迁移到服务端 cookie session。

### 2. 文件访问与隐私叙事尚未完全对齐

上传图/结果图的访问边界需要进一步收紧，否则不适合承接正式交付与售后找回。

建议方向：

- 鉴权下载接口
- 或带过期时间的签名 URL

### 3. 任务层不足以支撑履约与客服

仅靠 `task_id` 很难回答：

- 用户这单付了多少钱
- 承诺交付多少张
- 已交付多少张
- 哪些失败不算用户重拍
- 当前剩余额度还有多少

这些都必须上升到订单层。

## 讨论结论

“不想让用户登录”应被翻译为：

**不让用户感受到登录阻力**

而不是：

**没有身份、没有订单、没有售后主线**

最终方向：

- 首屏保持无登录
- 默认匿名身份承接流程
- 用订单作为支付、交付、售后的统一主键
- 在真正需要找回与跨设备时再轻绑定

## 后续建议动作

1. 先落 `orders`、`payment_transactions`、`generation_batches` 三张核心表
2. `POST /api/generate` 改为必须带 `order_id` 并校验权益
3. 审片页从“任务结果页”升级为“订单交付页”
4. 第二阶段补 `aftersale_tickets`
5. 第三阶段补邮箱或手机号轻绑定能力
