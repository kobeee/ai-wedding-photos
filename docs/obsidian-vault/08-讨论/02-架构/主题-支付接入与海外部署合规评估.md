# 主题：支付接入与海外部署合规评估

关联原案：[[支付接入方案]]

## 当前约束

用户当前的真实约束不是“理论上怎样最完整”，而是：

- 个人，且仍在职
- 希望尽快上线
- 服务当前部署在海外 VPS
- 需要兼顾安全与合法，不踩明显红线

因此，这次讨论重点不是“支付能力最全的方案”，而是“在当前约束下什么最能落地”。

## 当前站点与支付现状

现有产品形态：

- 前端已经展示 `¥0 / ¥99 / ¥599` 三档价格
- 站内流程是：上传 → 试妆 → 选视觉套餐 → 直接生成
- 代码里尚未引入订单、支付单、支付回调、退款和权益模型

因此，支付接入的本质不是“加一个微信/支付宝按钮”，而是补一层：

- SKU
- 订单
- 支付状态
- 回调验签
- 支付后权限解锁

## 支付方式可行性讨论

### 支付宝

当前结论：

- 在现有约束下，支付宝是更现实的第一选择
- 官方接入流程支持创建应用、配置密钥、开发与上线
- 从准入门槛看，比微信支付更有机会先跑通

这一轮讨论里的保守结论是：

- 如果要“快速上线”，优先做支付宝
- 不要一开始就把支付宝和微信双通道同时当作必选项

### 微信支付

当前结论：

- 微信支付的主体要求更严格
- 对当前“个人副业 + 海外部署 + 想尽快上”的约束并不友好

进一步拆分：

- `H5支付` 不适合作为第一期
- `JSAPI支付` 不适合作为第一期
- `Native支付` 只有在主体和应用载体齐备后才值得做

### 为什么不建议先做微信 H5 / JSAPI

这轮讨论中的关键判断：

- `微信 H5` 官方准入涉及支付域名配置与备案材料，和当前海外 VPS 的实际部署形态不匹配
- `微信 JSAPI` 适用于微信内浏览器闭环，通常还会引入公众号、openid、授权域名等额外复杂度
- 对当前项目来说，这两条都不是“最快形成收费闭环”的路径

所以更稳的判断是：

- 第一阶段不做微信 H5
- 第一阶段不做微信 JSAPI
- 以后如果主体、备案和私域链路都成熟，再考虑补齐

## 合规风险讨论

### 1. 真正的大风险不是支付，而是人脸数据

这一轮讨论里反复确认的一点是：

- 这个项目处理的是人脸照片
- 人脸/生物识别信息属于高敏感度个人信息
- 当前系统又是海外 VPS 存储与处理

因此，真正的风险优先级是：

1. 敏感个人信息处理
2. 跨境传输与境外存储
3. 用户授权与删除机制
4. 支付准入

也就是说，支付不是唯一门槛，甚至不是最大门槛。

### 2. 当前最需要的产品与工程补位

无论最终先上哪种支付方式，这些都应该先补：

- 隐私政策
- 用户协议
- 敏感个人信息单独同意
- 对“数据会被上传、处理、生成、临时存储、自动删除”的明确告知
- 删除周期与清理任务真正落地
- 客服/退款规则与履约说明

### 3. 关于海外 VPS 的判断

这轮讨论给出的工程判断是：

- 海外 VPS 本身不等于绝对不能运行
- 但“面向中国用户收集真实人脸并在海外处理”会提高合规压力
- 因此不适合在基础文档和授权都未补齐时直接大规模公开卖

更稳妥的产品策略：

- 先小范围、邀请制、灰度验证
- 先做最小收费闭环，不做大规模公开投放

## 当前讨论形成的结论

## 结论一：支付顺序应该重排

从“完整能力优先”改为“现实可落地优先”。

建议顺序：

1. 支付宝优先
2. 微信先缓
3. 若后续具备主体与应用条件，再考虑微信 Native
4. 微信 H5 / JSAPI 放更后面

## 结论二：支付前要先把订单层补出来

必须新增：

- `orders`
- `payment_transactions`
- `refunds`

并把 `/api/generate` 改成：

- 必须有合法 `order_id`
- 订单已支付或是明确的免费体验单
- 已退款 / 已关闭 / 已过期订单不能生成

## 结论三：当前阶段更适合“小闭环试运行”

建议策略：

- 支付能力最小化
- 上线对象最小化
- 宣传范围最小化
- 数据保留时间最小化

也就是：

- 先形成一个可以真实收款、真实生成、真实删除的闭环
- 再决定是否扩渠道和扩支付方式

## 结论四：不要把“快速上线”理解成“什么都先不管”

这轮讨论里达成的一致是：

- 快速上线不等于忽略安全和授权
- 至少要把最基本的边界写清楚、做出来

最低限度包括：

- 支付前展示用户协议与隐私政策
- 明确说明处理的是用户上传人像
- 明确说明数据会在海外基础设施或第三方 AI 服务中处理
- 明确说明自动删除时间
- 明确说明退款和不可交付规则

## 建议的执行顺序

### Phase 0：先补文档与授权边界

- 隐私政策
- 用户协议
- 敏感人脸信息授权说明
- 支付与退款说明

### Phase 1：只做最现实的收费闭环

- 先做支付宝
- 先补订单与回调
- 先把支付成功后才能生成这条闸门做好

### Phase 2：再决定是否扩大范围

满足下面条件后，再评估微信：

- 有更稳定的主体方案
- 有更明确的对外经营策略
- 有更稳的合规表达与数据处理边界

## 当前建议结论

> [!warning]
> 这不是正式法律意见，而是基于当前官方准入信息和工程现实给出的保守实施建议。

对当前这个项目，最现实的路线不是：

- 海外 VPS + 直接做微信 H5/JSAPI + 面向国内公开售卖

而是：

- 先把安全、授权、删除、订单层做实
- 先用支付宝打通最小收费闭环
- 以小范围验证代替大规模公开推广
- 把微信支付放到主体和条件更完整之后

## 后续待办

可以从这篇讨论继续拆成 3 个具体任务：

1. 支付宝优先的订单与支付接入实施方案
2. 隐私政策 / 用户协议 / 授权文案草案
3. 海外部署下的数据删除、回调安全与密钥管理清单
